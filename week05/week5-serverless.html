Serverless Backend — Lambda and API Gateway

Replacing EC2 and ALB with AWS Lambda and API Gateway. One architectural change resolves five Week 4 problems — and WAF closes the sixth.

Learning objectives
  • Explain why Lambda eliminates supply chain risk compared to EC2 with pip
  • Describe the Lambda execution model — invocation, cold start, timeout, and concurrency
  • Explain what a Lambda execution role is and how least-privilege applies to serverless functions
  • Describe what an API Gateway resource policy does and why it is stronger than the ALB custom header pattern
  • Explain where WAF sits in the request path and how a rate-based rule works
  • Describe what CloudWatch Logs captures automatically for every Lambda invocation
  • Explain why API Gateway is HTTPS-only and what that means for end-to-end encryption

1. Where we left off

Week 4 introduced a working dynamic backend — EC2, ALB, DynamoDB. But the architecture had six problems. Five are resolved by moving to Lambda and API Gateway. The sixth — no WAF — is added this week as a distinct step.

Issue 1 — No WAF

CloudFront had no Web Application Firewall. Any IP could flood /photos with unlimited requests — CloudFront forwarded every one to the ALB and EC2. Shield Standard only protects against L3/L4 volumetric attacks; valid HTTP requests pass straight through. Fixed this week by attaching a WAF Web ACL to CloudFront with a rate-based rule on /photos.

Issue 2 — ALB directly reachable

The ALB had a public DNS name reachable by any IP — not just CloudFront. An attacker who found the ALB DNS name could call /photos directly, bypassing CloudFront's WAF, geo-restrictions, and edge controls. Fixed this week by replacing the ALB with API Gateway, which supports a resource policy restricting invocation to your specific CloudFront distribution ARN — IAM-enforced, not header-matching.

Issue 3 — Supply chain risk

The EC2 user data script ran pip3 install boto3 at boot — pulling from PyPI over a NAT Gateway. A malicious package would run as root with the EC2 IAM role attached. Fixed this week by moving to Lambda. Lambda provides the Python runtime and boto3 with no installation step. There is no NAT Gateway, no PyPI call, no opportunity for a malicious package to enter.

Issue 4 — EC2 operational burden

EC2 ran continuously at ~$20–25/month idle cost. Every instance required OS patching, SSM agent updates, and Amazon Linux version management — obligations students rarely fulfil in a lab environment, leaving known-vulnerable OS versions running. Fixed this week by Lambda. Lambda runs only on invocation — idle cost is zero. AWS manages the runtime entirely, including security patches.

Issue 5 🔶 — CloudFront → ALB hop was HTTP

We deliberately skipped provisioning an ACM certificate for the ALB, so the CloudFront → ALB hop ran over HTTP (port 80). Traffic was in plaintext on AWS's internal network — low practical risk but not defense-in-depth. Fixed this week automatically — API Gateway only exposes an HTTPS endpoint. No certificate to manage.

Issue 6 🔶 — Single EC2 instance in one AZ

We deliberately ran one EC2 instance in one AZ. The ALB spanned two AZs but had only one target. A crash or reboot took /photos offline with no failover — manual recovery via SSM, 30–90 seconds of 502s. Fixed this week automatically — Lambda is inherently multi-AZ. AWS runs function instances across multiple AZs with no configuration required.

2. This week's solution

What serverless means

In Week 4, the backend ran on EC2 — a virtual server that ran continuously whether or not requests were coming in. You were responsible for the OS, the Python runtime, patching, and keeping the process alive via systemd. Serverless flips this model. You provide only the function code; AWS handles everything below it — the servers, OS, runtime, scaling, and availability. The function runs only when invoked and stops when it returns. There is no persistent process, no idle cost, and no infrastructure to maintain.

Security benefits of serverless: No OS to patch — no CVE exposure from an unpatched Amazon Linux instance. No runtime to install — no supply chain risk from pip or npm. No persistent process — no long-lived server to compromise. Automatic CloudWatch Logs on every invocation — a full audit trail with zero configuration. Five of the six Week 4 vulnerabilities are resolved by this single architectural shift.

Two changes this week: replace the EC2/ALB backend with Lambda and API Gateway, and attach WAF to CloudFront. The DynamoDB table and CloudFront distribution from Week 4 are reused unchanged.

The 3-tier pattern is preserved — but the application tier is now serverless. Click any component to learn what it does.

Security principle: Each tier only communicates with the tier immediately adjacent to it. The browser never reaches Lambda or DynamoDB directly. Lambda never talks to the browser directly. WAF and API Gateway enforce this containment — WAF at the edge, API Gateway at the application entry point.

3. Architecture — AWS service boundary

All services this week are fully managed — there is no VPC. The diagram shows the AWS global service boundary and contrasts it with the VPC infrastructure Week 4 required. Click any service to learn more.

Click any component to explore

Key architecture points

No VPC

Lambda, API Gateway, WAF, CloudFront, S3, and DynamoDB are all fully managed services. There are no subnets to design, no security groups to configure, no route tables to manage. AWS handles the network infrastructure entirely.

Two origins

CloudFront serves as the single entry point for all traffic. It routes /photos requests to API Gateway (CachingDisabled) and all other requests to S3 via OAC. The browser only ever talks to CloudFront.

HTTPS end-to-end

Every hop is encrypted. Browser → CloudFront → API Gateway — all HTTPS. Unlike Week 4 where the CloudFront → ALB hop was HTTP, API Gateway only exposes HTTPS endpoints. No certificate management required.

IAM enforcement

The API Gateway resource policy uses aws:SourceArn to restrict invocations to our specific CloudFront distribution. IAM-enforced, not header matching. DynamoDB is reached over AWS's private network — no internet path.

4. How it works — request walkthrough

From the moment a user opens the Photo Viewer to the moment photos appear on screen — every component plays a role.

Step 1
Browser requests the page

User opens the Photo Viewer URL. The browser hits CloudFront at the nearest edge POP. CloudFront forwards to S3 via OAC and returns index.html, style.css, and app.js — served from cache if available.

Step 2
JavaScript requests the photo list

app.js makes a fetch('/photos') call. CloudFront has a cache behavior for this path with CachingDisabled — it routes the request to the API Gateway origin, not S3. Every /photos call goes through.

Step 3
WAF evaluates the request

Before CloudFront routes to the origin, WAF checks the rate-based rule for /photos. It counts requests from this source IP over the last 5 minutes. Under the threshold → continues. Over the threshold → WAF returns 429 immediately, Lambda is never invoked.

Step 4
CloudFront forwards to API Gateway

CloudFront connects to API Gateway over HTTPS — unlike Week 4 where the origin hop was HTTP. No port 80, no plaintext. API Gateway receives the request with CloudFront metadata in the headers.

Step 5
API Gateway checks the resource policy

API Gateway evaluates the resource policy. It checks aws:SourceArn — the request must originate from our specific CloudFront distribution ARN. Match → Lambda is invoked. No match (e.g. a direct call to the execute-api URL) → 403 Forbidden, Lambda is never invoked.

Step 6
Lambda executes and queries DynamoDB

AWS invokes lambda_function.py. If the execution environment is cold, Python initializes (~100–300ms). The handler runs table.scan() with FilterExpression is_public = true. DynamoDB traffic stays on AWS's private network — no internet path. The invocation is logged to CloudWatch automatically.

Step 7
Lambda returns JSON up the chain

Lambda returns [{"photo_id":"photo-001","s3_key":"photos/photo1.jpg"}, ...] to API Gateway, which returns it to CloudFront, which returns it to the browser.

Step 8
Browser fetches and displays photos

app.js loops through the JSON and requests each photo image from CloudFront → S3 via OAC. Same path as Week 3. Photos appear on screen.

Verify in DevTools: Open your Photo Viewer site in the browser. Press F12 (or right-click → Inspect) to open DevTools, then go to the Network tab. Reload the page and click the /photos request. In the Response Headers you should see Via: CloudFront — confirming the request went through CloudFront — and no Server: Werkzeug header. That header is gone because there is no longer a Flask server. The response came from Lambda via API Gateway.

5. Key concepts

Click any row to expand

λAWS LambdaServerless compute · invocation model · cold start · concurrency
›

Lambda is a serverless compute service — you provide a function, AWS provides everything else: servers, OS, runtime, scaling, patching, availability. You are billed only for the time your code actually runs — idle cost is zero. There is no persistent process: each invocation gets its own isolated execution environment, which is discarded after the function returns.

Execution model: When a request arrives, AWS either reuses a warm environment or creates a new one — a cold start (typically 100–500ms for Python). After the function returns the environment is held briefly for possible reuse, then discarded.

Key parameters:

  • Timeout — maximum execution time (default 3 seconds, max 15 minutes). Function exceeds timeout → Lambda terminates it and returns 504.
  • Memory — allocated RAM (128MB–10GB). CPU allocation scales proportionally.
  • Concurrency — each simultaneous request gets its own environment. Lambda scales horizontally automatically. Default account limit: 1,000 concurrent executions per region.

In the demo: lambda_function.py handles GET /photos — scans DynamoDB, returns JSON. No Flask, no port to listen on, no server to manage.

🔌Amazon API GatewayManaged API layer · HTTP API · HTTPS-only · built-in throttling
›

API Gateway is a fully managed service that handles the HTTP layer for your backend — routing, throttling, authentication, access control — so your Lambda function only handles business logic. Unlike the ALB in Week 4, API Gateway is IAM-aware: you can attach resource policies, Cognito authorizers, and Lambda authorizers directly to the API.

HTTP API vs REST API: API Gateway offers two types. REST API has more features (usage plans, API keys, per-method throttling). HTTP API is simpler, cheaper, and lower latency — sufficient for our use case. We use HTTP API.

Built-in throttling: API Gateway enforces default account-level throttling (10,000 requests/second burst, 5,000 requests/second steady-state) independently of WAF. This is a backstop against runaway traffic even before WAF rate rules activate.

HTTPS only: API Gateway exposes HTTPS endpoints exclusively. There is no HTTP option — TLS is enforced by the service, not by configuration.

In the demo: HTTP API with one route: GET /photos → lambda_function. Resource policy restricts invocations to our CloudFront distribution ARN.

🛡️AWS WAF — Web Application FirewallL7 inspection · Web ACL · rules and rule groups · attached to CloudFront
›

AWS WAF is a web application firewall that inspects HTTP requests at Layer 7 — the application layer. Unlike network firewalls that filter by IP and port, WAF reads the content of requests: the path, headers, query strings, and body. This lets it detect and block attacks that look like legitimate traffic at the network layer.

Web ACL: A Web Access Control List is the top-level WAF resource. You attach it to a CloudFront distribution (or an API Gateway stage, ALB, or AppSync API). The ACL contains an ordered list of rules evaluated against every request.

Rule types:

  • Rate-based rules — count requests per IP over a time window, block when a threshold is exceeded
  • Managed rule groups — curated rule sets maintained by AWS or third parties: Core Rule Set (OWASP Top 10), Known Bad Inputs, SQL injection, XSS, Bot Control. Many are free; some are paid.
  • Custom rules — match on specific strings, IP sets, geo conditions, or combinations using AND/OR logic

WAF vs Shield Standard: Shield Standard (free, automatic) protects against L3/L4 volumetric attacks — SYN floods, UDP reflection. WAF operates at L7 and catches application-layer abuse that Shield cannot see. They complement each other.

In the demo: Web ACL attached to our CloudFront distribution with one rate-based rule scoped to the /photos path.

Security controls

Click any row to expand

🔑Lambda execution roleIAM role · least privilege · temporary credentials · no access keys
›

Every Lambda function has an execution role — an IAM role that AWS assumes on the function's behalf for each invocation. Lambda automatically injects temporary credentials (access key, secret key, session token) into the execution environment. These credentials expire and rotate automatically. There are no long-lived access keys to store, rotate, or accidentally leak.

Our role — least privilege:

  • dynamodb:Scan on arn:aws:dynamodb:us-east-1:644094189785:table/photoviewer-photos only — no other tables, no writes
  • logs:CreateLogGroup, logs:CreateLogStream, logs:PutLogEvents — required for CloudWatch Logs

No dynamodb:*, no S3, no IAM. If the function is compromised, the blast radius is one table scan. Notice that lambda_function.py contains no credentials — the execution role handles all authentication to AWS services automatically.

📜API Gateway resource policyIAM-enforced · restricts to CloudFront ARN · stronger than custom headers
›

A resource policy is an IAM policy attached directly to the API. It controls who can call execute-api:Invoke. We use the aws:SourceArn condition key to restrict invocations to our specific CloudFront distribution ARN — so only requests originating from our distribution are allowed through.

Why this is stronger than the ALB custom header pattern:

  • Custom headers are string matching — any caller who knows the secret passes. The secret lives in CloudFront's origin config in plaintext and must be manually rotated.
  • Resource policy uses IAM conditions — enforced cryptographically by AWS's authorization layer before the request reaches your API. There is no secret to discover or leak.
  • The aws:SourceArn condition ties to your specific distribution ARN — not just "any CloudFront IP," which is what the ALB security group prefix list gave us.
⚡WAF rate-based ruleRequests per IP · rolling window · 429 response · Lambda never invoked
›

We attach a Web ACL to the CloudFront distribution with one rate-based rule scoped to the /photos path. WAF evaluates every request before CloudFront routes it — so a blocked request never reaches API Gateway or Lambda.

How it works: WAF counts requests per source IP over a rolling 5-minute window. If a single IP exceeds the configured threshold, WAF blocks subsequent requests from that IP with a 429 response. The block lifts automatically once the request rate drops below the threshold.

What this stops: A script hammering /photos — consuming Lambda invocations and DynamoDB read capacity — is stopped at the edge before it costs you anything. Real users are unaffected as long as their request rate is below the threshold.

What this doesn't stop: A distributed flood from many IPs simultaneously (each below the per-IP threshold). That requires managed rule groups or Bot Control — beyond this week's scope but worth knowing.

📋CloudWatch Logs — automatic audit trailEvery invocation logged · no agent · detective control
›

Every Lambda invocation is automatically logged to CloudWatch Logs — no agent to install, no configuration required beyond the execution role having logs:PutLogEvents. Lambda creates a log group (/aws/lambda/photoviewer-lambda) and a new log stream per execution environment.

What's logged automatically per invocation:

  • START — invocation ID, runtime version
  • END — invocation ID
  • REPORT — duration (ms), billed duration (ms), memory used, cold start indicator
  • Any print() or logging.* output from your function code

Security classification: CloudWatch Logs is a detective control — it doesn't prevent attacks but creates a record that enables detection and investigation after the fact. Compare this to EC2: in Week 4, logging required manually installing and configuring the CloudWatch agent. With Lambda, the audit trail is automatic and non-bypassable.

🔐End-to-end HTTPSEncryption in transit · API Gateway enforces TLS · no certificate to manage
›

In Week 4, the CloudFront → ALB hop ran over HTTP (port 80) because we skipped provisioning an ACM certificate for the ALB. That left one hop unencrypted — a deliberate lab simplification that violates defense-in-depth.

API Gateway exposes HTTPS endpoints only. There is no HTTP option — TLS is enforced by the service. AWS manages the certificate for execute-api.amazonaws.com automatically. The result is true end-to-end encryption with no configuration on our part: browser → CloudFront → API Gateway, all HTTPS.

Security classification: This is a preventive control — it prevents a man-in-the-middle attacker from reading or injecting content into the CloudFront → origin hop. It operates on the CIA triad's Confidentiality and Integrity dimensions.

6. What's still missing

Click each problem to see the impact and what comes next.

No authentication — /photos is still public
click to expand

The limitation: Any request to /photos returns the full list of public photos — no login, no identity check. API Gateway passes every request to Lambda regardless of who is asking. The WAF rate-based rule limits flooding but doesn't establish identity.

Why it matters: The DynamoDB table already has an owner field and is_public flag — the data model supports private photos and per-user access. But without knowing who the caller is, Lambda can only return the public set for everyone. There is no way to build a free tier, a premium tier, or user-specific photo management without identity.

What's next: Week 6 adds Amazon Cognito — a user pool for registration and login. API Gateway gets a Cognito authorizer that validates the JWT token on every request before Lambda runs. Lambda then knows who is calling and can filter accordingly.

API Gateway endpoint has a public URL
click to expand

The limitation: The API Gateway endpoint (https://XXXXXX.execute-api.us-east-1.amazonaws.com/photos) has a publicly resolvable URL. The resource policy restricts invocations to our CloudFront distribution ARN — so a direct call without the correct CloudFront origin is denied at the IAM layer. But the URL is discoverable, and any misconfiguration of the resource policy would immediately expose the API.

The distinction from Week 4: The ALB was reachable by any IP that knew the DNS name — no policy enforcement. API Gateway with a resource policy requires the request to originate from our specific CloudFront distribution, enforced by IAM. The exposure surface is much smaller — but the URL being public is worth noting.

Production hardening: In production, you can attach a second WAF Web ACL directly to the API Gateway stage (Regional WAF, separate from the CloudFront WAF). This means even direct calls to the execute-api URL — which the resource policy rejects — are rate-limited before consuming API Gateway capacity. Two WAF layers: one at the CloudFront edge catching floods early, one at API Gateway as a backstop if CloudFront is misconfigured or bypassed. For this course, the CloudFront WAF combined with the resource policy is the correct and sufficient posture.
