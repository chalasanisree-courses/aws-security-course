Week 5 Lab — Serverless Backend

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  100 pts

Objective: Replace the EC2 and ALB backend from Week 4 with AWS Lambda and API Gateway. Add AWS WAF to protect the /photos endpoint. By the end of this lab, the Photo Viewer backend is fully serverless — no servers to patch, no ALB to misconfigure, end-to-end HTTPS, and WAF rate limiting at the edge.

Browser → WAF/CloudFront → API Gateway → Lambda → DynamoDB ↘ S3 (static files via OAC — unchanged)
WAF runs at the CloudFront edge POP, evaluating requests before CloudFront routes to any origin. A WAF-blocked request never reaches API Gateway or Lambda.
Before you start: Your Week 4 CloudFront distribution must be working. Have your CloudFront distribution ID and domain name ready — you will need them for the API Gateway resource policy and WAF configuration. Your DynamoDB table photoviewer-photos must still have the 5 photo items seeded in Week 4.
Files for this lab: Get all files from the course GitHub repository → photoviewer/week05

Part 1 — Lambda Function

Step 1Create the Lambda function

Go to Lambda → Create function → Author from scratch:

  • Function name: photoviewer-lambda
  • Runtime: Python 3.12
  • Architecture: x86_64
  • Execution role: Create a new role with basic Lambda permissions
  • Click Create function
AWS creates the function and an IAM role named photoviewer-lambda-role-XXXXXXXX. The auto-created role includes CloudWatch Logs permissions only — you will add DynamoDB access in the next step.
Step 2Add DynamoDB permission to the execution role

Go to Configuration tab → Permissions → click the role name to open it in IAM.

Click Add permissions → Create inline policy → JSON tab and paste:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["dynamodb:Scan"],
    "Resource": "arn:aws:dynamodb:us-east-1:[your-account-id]:table/photoviewer-photos"
  }]
}

Policy name: photoviewer-dynamodb-read → Create policy

Verify the role now shows two policies:

  • AWSLambdaBasicExecutionRole — CloudWatch Logs (auto-created)
  • photoviewer-dynamodb-read — DynamoDB Scan on one table
Screenshot: IAM role showing both policies — AWSLambdaBasicExecutionRole and photoviewer-dynamodb-read — with the DynamoDB policy expanded to show the Scan action on the correct table ARN
Step 3Deploy lambda_function.py

Go back to your Lambda function → Code tab. You will see a default lambda_function.py with a placeholder. Replace the entire contents with the code from the Week 5 GitHub repo.

Click Deploy to save and activate the new code.

The function uses Attr('is_public').eq(True) to filter the DynamoDB scan — the same logic as app.py in Week 4 but without Flask, ports, or a server process. No packages need to be installed — boto3 is provided by the Lambda runtime automatically.
Step 4Test the Lambda function in isolation

Go to Test tab → Create new test event:

  • Event name: test-photos
  • Template: API Gateway HTTP API (or leave as hello-world — Lambda ignores the event body for this function)
  • Click Test

The execution result should show:

  • Status code: 200
  • Body: a JSON array of 5 photo objects from DynamoDB
If you see an error: Check the execution role has dynamodb:Scan on the correct table ARN with your account ID. Check that the DynamoDB table photoviewer-photos exists in us-east-1.
Screenshot: Lambda test result showing status 200 and the JSON array of photos in the response body

Part 2 — API Gateway

Step 5Create HTTP API with Lambda integration

Go to API Gateway → Create API → HTTP API → Build:

  • Click Add integration → select Lambda → select photoviewer-lambda
  • API name: photoviewer-api
  • Click Next

On the Configure routes screen:

  • Method: GET
  • Resource path: /photos
  • Integration target: photoviewer-lambda
  • Click Next

On the Define stages screen: leave $default with Auto-deploy enabled → Next → Create.

Copy the Invoke URL — it looks like https://XXXXXXXXXX.execute-api.us-east-1.amazonaws.com. You will need the hostname (without https://) for the CloudFront origin.

Screenshot: API Gateway showing the GET /photos route and the Invoke URL
Step 6Add resource policy — restrict to CloudFront only

Go to your API → Settings (left menu) → Resource policy. Paste the following, replacing both placeholders with your values:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "execute-api:Invoke",
      "Resource": "arn:aws:execute-api:us-east-1:[your-account-id]:*/*/GET/photos",
      "Condition": {
        "StringEquals": {
          "aws:SourceArn": "arn:aws:cloudfront::[your-account-id]:distribution/[your-distribution-id]"
        }
      }
    }
  ]
}

Click Save. Then go to Deploy (top right) to redeploy the API with the policy active.

Your CloudFront distribution ID is on the CloudFront console — it looks like EXXXXXXXXXXXXXXXXX. The resource policy enforces that only requests originating from your specific CloudFront distribution can invoke the API. Direct calls to the execute-api URL will return 403.
Screenshot: Resource policy JSON showing your CloudFront distribution ARN in the aws:SourceArn condition
Step 7Verify resource policy is enforced

Open a new browser tab and navigate directly to your API Gateway invoke URL:

https://XXXXXXXXXX.execute-api.us-east-1.amazonaws.com/photos

You should receive a 403 Forbidden response — confirming the resource policy is blocking direct access. The API will only respond to requests arriving through your CloudFront distribution.

Screenshot: Browser showing 403 response when accessing the API Gateway URL directly

Part 3 — Update CloudFront

Step 8Add API Gateway as a CloudFront origin

Go to CloudFront → your distribution → Origins tab → Create origin:

  • Origin domain: paste the API Gateway hostname only — XXXXXXXXXX.execute-api.us-east-1.amazonaws.com (no https://, no path)
  • Protocol: HTTPS only
  • Origin name: photoviewer-apigw-origin
  • Click Create origin
Unlike Week 4 where the CloudFront → ALB hop was HTTP (port 80), this origin uses HTTPS only. API Gateway only exposes HTTPS — there is no HTTP endpoint to configure. End-to-end encryption is automatic.
Step 9Update the /photos cache behavior

Go to Behaviors tab → find the /photos behavior → Edit:

  • Origin and origin groups: change from photoviewer-alb-origin to photoviewer-apigw-origin
  • Viewer protocol policy: Redirect HTTP to HTTPS
  • Cache policy: CachingDisabled
  • Click Save changes

Go to Invalidations tab → Create invalidation → path: /* → Create invalidation. Wait for status: Completed.

Part 4 — Clean Up Week 4 Infrastructure

Step 10Delete ALB and EC2
Do this before verifying. The ALB and EC2 are no longer needed — Lambda handles all backend requests. If left running, the old /photos behavior may still route to the ALB while you update CloudFront. ALB costs ~$16/month; EC2 costs ~$8/month at idle.
  1. Delete ALB: EC2 → Load Balancers → select photoviewer-alb → Actions → Delete load balancer → confirm
  2. Terminate EC2: EC2 → Instances → select photoviewer-ec2 → Instance state → Terminate instance → confirm
  3. Delete target group: EC2 → Target Groups → select photoviewer-tg → Actions → Delete
Keep the VPC, DynamoDB table, S3 bucket, CloudFront distribution, and IAM roles — these are still needed. The EC2 IAM role (photoviewer-ec2-role) can be kept for reference or deleted.
Screenshot: EC2 instances list showing photoviewer-ec2 in terminated state, and Load Balancers list showing no active load balancers

Part 5 — WAF

Step 11Create WAF Web ACL
Region matters: WAF for CloudFront must be created in us-east-1 (N. Virginia). Verify the region selector in the top right of the AWS console shows N. Virginia before proceeding. If you create WAF in another region, it will not be available to associate with CloudFront.

Go to WAF & Shield → Web ACLs → Create web ACL:

  • Resource type: Amazon CloudFront distributions
  • Name: photoviewer-waf
  • Associated AWS resources: click Add AWS resources → select your CloudFront distribution → Add
  • Click Next
Step 12Add rate-based rule

On the Add rules and rule groups screen:

  • Click Add rules → Add my own rules and rule groups → Rule builder
  • Rule type: Rate-based rule
  • Name: photos-rate-limit
  • Rate limit: 100
  • Evaluation window: 5 minutes
  • Scope of inspection: Only consider requests that match the criteria below
  • Criteria: URI path → starts with → /photos
  • Action: Block
  • Click Add rule

On the Set rule priority screen: leave the default order → Next.

On Configure metrics: leave defaults → Next.

On Review and create: verify the rule and associated CloudFront distribution → Create web ACL.

A rate of 100 requests per 5 minutes is appropriate for a lab. In production you would set this based on expected legitimate traffic patterns — typically 1,000–10,000 per 5 minutes for a public API.
Screenshot: WAF Web ACL showing the photos-rate-limit rule and the associated CloudFront distribution

Part 6 — Verify End to End

Step 13Test in browser

Open your Photo Viewer site using your CloudFront domain name (e.g. https://d1kbm2nphud61r.cloudfront.net). The photos should load exactly as in Week 4 — the user experience is unchanged.

Press F12 to open DevTools → go to the Network tab → reload the page → click the /photos request.

In the Response Headers panel confirm:

  • Via: 1.1 XXXXXXXXXX.cloudfront.net (CloudFront) — request went through CloudFront ✓
  • No Server: Werkzeug header — Flask on EC2 is gone ✓
  • x-amzn-requestid — present, confirming the response came from API Gateway / Lambda ✓
Expected result: Photos load correctly and DevTools confirms the response is no longer coming from Flask/EC2.
Screenshot: Photo Viewer loading in browser + DevTools Network tab showing the /photos request with Via: CloudFront header and no Server: Werkzeug header
Step 14Verify Lambda invocations in CloudWatch Logs

Go to CloudWatch → Log groups → /aws/lambda/photoviewer-lambda.

Click the most recent log stream. You should see entries for each invocation including:

  • START — invocation ID and runtime version
  • END — invocation ID
  • REPORT — duration in ms, billed duration, memory used

Count the log entries — they should correspond to the number of times you loaded the Photo Viewer.

Every /photos request that reaches Lambda is automatically logged here — no configuration required. This is your audit trail for the API. Note that WAF-blocked requests never reach Lambda and will not appear in these logs — they are captured in WAF logs instead.
Screenshot: CloudWatch log stream showing START / END / REPORT entries for Lambda invocations
Step 15Verify WAF is attached

Go to WAF & Shield → Web ACLs → photoviewer-waf → Associated AWS resources tab.

Confirm your CloudFront distribution is listed as an associated resource.

Go to the Overview tab — after loading the Photo Viewer a few times you should see allow counts appearing in the traffic overview chart (it may take 1–2 minutes to update).

Screenshot: WAF Web ACL Associated AWS resources tab showing your CloudFront distribution, and the Overview tab showing allow counts

Reflection

Answer all three questions in 2–3 sentences each.

Question 1

In Week 4, the ALB security group allowed inbound HTTP on port 80 from 0.0.0.0/0 — any IP could reach it directly, bypassing CloudFront entirely. In Week 5, the API Gateway resource policy uses aws:SourceArn to restrict access to your specific CloudFront distribution ARN. Explain why the aws:SourceArn approach is more secure than the open security group, and why even a CloudFront IP prefix list on the ALB security group would still be weaker than the resource policy approach.

Question 2

Week 4 required a NAT Gateway so EC2 could download packages from the internet during bootstrap. Week 5 has no NAT Gateway and no package installation step. Explain why Lambda eliminates the need for a NAT Gateway and what security benefit this provides — specifically in terms of supply chain risk.

Question 3

The WAF rate-based rule and the API Gateway resource policy both protect the /photos endpoint, but they protect against different threats. Explain what each one does and why both are necessary — what does WAF stop that the resource policy does not, and vice versa?
