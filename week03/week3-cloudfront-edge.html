CloudFront and Edge Security

How Amazon CloudFront fixes the three Week 2 vulnerabilities — and the additional controls that make the edge a real security boundary.

Learning objectives
  • Explain what a CDN is and how CloudFront caches content at the edge
  • Describe what Origin Access Control does and why it's needed
  • Explain the difference between the S3 REST endpoint and the S3 website endpoint
  • Describe why bucket enumeration is structurally impossible through CloudFront
  • Explain how CloudFront Functions enforce an allowlist at the viewer request stage
  • Describe what CloudFront policies control and where each one sits in the request flow
  • Identify what Strict-Transport-Security does and when it matters

1. Where we left off

Week 2 hosted the Photo Viewer directly from S3 static website hosting. It worked — but introduced three security and performance problems that make it unsuitable for a real application.

Issue 1 — HTTP only

The S3 website endpoint is HTTP only — TLS is not supported at this endpoint. All content travels in plaintext. Anyone on the same network can read it (man-in-the-middle). For a photo gallery this seems low risk — but the same pattern carries forward into later weeks where users will submit credentials and upload private photos.

Issue 2 — Public bucket

Static website hosting requires Block Public Access to be disabled and a policy granting Principal: *. Every file in the bucket is publicly accessible — not just web files. confidential.txt is readable by anyone who knows the path. And because s3:ListBucket was granted, anyone can enumerate the bucket with aws s3 ls --no-sign-request.

Issue 3 — No caching

The S3 bucket lives in us-east-1. Every request from every user — wherever they are — travels to Northern Virginia and back. For a real application serving users globally this latency is unacceptable and every request costs money.

2. This week's solution

We place CloudFront — Amazon's global CDN — in front of S3. This single change addresses all three Week 2 problems simultaneously, and adds several additional security controls at the edge.

3. Architecture

Week 3 inserts CloudFront between the browser and S3. The bucket goes fully private — only CloudFront can read it via OAC. Every request passes through CloudFront's edge before anything reaches S3.

5. Key concepts

🌐What is a CDN?Edge locations · cache hits and misses · why latency matters
›

A Content Delivery Network (CDN) is a globally distributed network of servers — called edge locations or Points of Presence (POPs) — that store copies of your content close to your users. Instead of every request traveling to your origin server, users are served from the nearest edge location.

Why distance causes latency: Data travels at roughly two-thirds the speed of light through fiber. A request from Tokyo to a server in Virginia travels ~10,000km each way — adding 150–200ms of round-trip latency before a single byte is sent. For video streaming that means buffering. For a web page it means a visibly slow load. For a global application it means users in far regions get a worse experience than users near your origin.

Cache hit vs cache miss:

EventWhat happensOrigin contacted?
Cache hitEdge has a fresh copy — serves it immediatelyNo
Cache missEdge has no copy or TTL expired — fetches from originYes — once, then cached

After a cache miss, the edge stores the response for the TTL duration. All subsequent requests get a cache hit until TTL expires. For a static photo gallery almost every request after the first warm-up will be a cache hit.

›

CloudFront is a globally distributed cache. Requests are served from the nearest edge location — a user in Tokyo gets content from a Tokyo POP, not Northern Virginia. CloudFront also terminates HTTPS at the edge, replacing the HTTP-only S3 website endpoint entirely.

CloudFront reaches S3 via the REST endpoint — not the static website endpoint. The REST endpoint supports TLS and authenticated requests (required for OAC). The website endpoint supports neither.

Bucket enumeration is structurally impossible through CloudFront. CloudFront only issues GetObject requests for specific keys — there is no HTTP operation that maps to s3:ListBucket. The aws s3 ls --no-sign-request command that worked in Week 2 fails in Week 3 because it talks directly to the S3 API, which is now locked by OAC.

In the demo

We disabled S3 static website hosting, turned Block Public Access back on, and created a CloudFront distribution pointing to the bucket. Once deployed, the distribution URL (d1xxxxxxxxxx.cloudfront.net) became the only way to reach the Photo Viewer — direct S3 access returns 403. The bucket policy was updated to allow s3:GetObject only from our specific CloudFront distribution ARN.

🔑Origin Access Control (OAC)SigV4 signing · bucket policy locks to your distribution ARN only
›

Adding CloudFront in front of S3 does not automatically make the bucket private — without OAC, anyone can bypass CloudFront and hit S3 directly. OAC uses two parts that must work together:

Part 1 — Request signing: CloudFront signs every request to S3 using AWS Signature Version 4. S3 verifies the signature proves it came from your specific distribution.

Part 2 — Bucket policy: Only allows s3:GetObject from requests signed by your distribution ARN. Everything else — direct S3 URL, CLI, any other tool — gets 403.

{
  "Principal": {"Service": "cloudfront.amazonaws.com"},
  "Action": "s3:GetObject",
  "Condition": {"StringEquals": {
    "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT:distribution/DIST_ID"
  }}
}

What OAC does not fix: OAC controls who can talk to S3. A request for /confidential.txt through CloudFront would still be served unless blocked by a CloudFront Function.

In the demo

After creating the CloudFront distribution, we copied the OAC-generated bucket policy into S3 — replacing the old Principal: * policy. Block Public Access was turned back on. Direct S3 access then returned 403 for all requests, confirming only CloudFront could reach the bucket.

⚡CloudFront Function — allowlistViewer Request stage · blocks non-web files before cache · fails closed
›

Runs at the Viewer Request stage — before cache lookup or origin contact. Returns 403 immediately for blocked files, without ever touching S3.

function handler(event) {
    var uri = event.request.uri;
    var allowed = /\.(html|js|css|jpg)$/i;
    if (uri === '/' || allowed.test(uri)) return event.request;
    return { statusCode: 403, statusDescription: 'Forbidden' };
}

Allowlist vs blocklist: An allowlist permits only what is explicitly listed — everything else is blocked. A blocklist blocks only what is listed — everything else is permitted. Allowlists are more secure: a forgotten file type causes a blocklist to fail open, an allowlist to fail closed.

In the demo

A function called PhotoViewer-AllowList was created and associated with the distribution behavior at the Viewer Request stage. Accessing /confidential.txt through CloudFront returned 403 — the function blocked it before the request reached S3. We then removed the function, ran an invalidation, and accessed /confidential.txt again — this time it returned the file successfully, demonstrating exactly why the function is necessary.

📋CloudFront policiesViewer Protocol · Cache · Origin Request · Response Headers (HSTS)
›
PolicyStageWhat it controls
Viewer ProtocolBrowser ↔ CFRedirect HTTP to HTTPS automatically — if user types http://, CloudFront redirects before serving any content
Cache PolicyCF cacheTTL — how long objects stay cached. Cache hit → served immediately, no origin request. Always invalidate (/*) after security-relevant changes
Origin RequestCF → S3Controls what travels from CF to S3 on a cache miss. HTTP request headers can carry malicious payloads — strip headers, cookies, and query strings that S3 doesn't need. Least privilege for HTTP forwarding
Response HeadersCF → BrowserAdds security headers to every response — HSTS tells the browser to always use HTTPS on future requests, eliminating the first-request vulnerability window

Why HSTS matters even with HTTP redirect: The Viewer Protocol Policy redirects HTTP to HTTPS at CloudFront — but that first request from the browser still travels to CloudFront over HTTP before the redirect. In Week 6 when we add a login page, a user typing http:// and submitting their password would send credentials in plaintext before the redirect kicks in. HSTS fixes this: after the first HTTPS response, the browser enforces HTTPS itself on all subsequent requests — never sending the first request over HTTP again. We add it now as infrastructure baseline.

In the demo
  • Viewer Protocol Policy — set to Redirect HTTP to HTTPS. Typing the HTTP URL in the browser automatically redirected to HTTPS.
  • Cache Policy — set to CachingOptimized, the AWS-managed policy recommended for S3 origins.
  • Origin Request Policy — left as none for the Photo Viewer. S3 is a static origin — no headers, cookies, or query strings need to be forwarded.
  • Response Headers Policy — set to SecurityHeadersPolicy, the AWS-managed policy that adds HSTS, X-Frame-Options, and X-Content-Type-Options. Inspect these in browser DevTools → Network → any Photo Viewer response → Response Headers.
🌏Geo-restrictionBlock or allow by country · no WAF required · distribution-level only
›

Geo-restriction lets you control which countries can access your CloudFront distribution — no WAF, no Lambda, no extra cost. CloudFront determines the viewer's country from their IP address and enforces the restriction at the nearest edge POP before any content is served.

ModeHow it worksUse when
AllowlistOnly listed countries can accessApp is licensed for specific markets only
BlocklistListed countries are blocked, all others can accessCompliance requires blocking specific regions

Blocked requests receive a 403 at the nearest POP — the viewer never reaches your origin.

Limitation: Applies to the entire distribution. For path-level control (e.g. block one country from /admin only), use a CloudFront Function instead.

In the demo

Geo-restriction was configured in the demo by adding Iran to the blocklist under CloudFront → Security → Geographic Restrictions. This took effect within minutes with no changes to S3 or application code — a single CloudFront setting blocked an entire country at the edge. The restriction was added purely to demonstrate the feature; it is not active on your distribution for the lab.

🔄Cache invalidationForce immediate edge refresh · always run after security changes
›

After updating content in S3 — patched JavaScript, new CF Function code, updated policies — edge caches may still serve the old version until TTL expires. Create an invalidation to purge immediately: use /* for everything or specific paths like /app.js for targeted invalidation.

Always invalidate after security-relevant changes. Without invalidation, stale cached responses — missing your new headers, function behavior, or patched code — continue to be served globally until TTL expires.

In the demo

We ran an invalidation with /* after removing the CloudFront Function from the distribution behavior — to demonstrate what happens without the function. Without the invalidation, edge caches would have continued serving responses under the old function rules. After invalidation, accessing /confidential.txt returned the file successfully — confirming the function was no longer blocking it. We then re-associated the function to restore the protection.

6. What's still missing

Week 3 resolves all three Week 2 problems. For a static site, this architecture has no significant remaining security gaps — CloudFront with OAC, HTTPS enforcement, response security headers, and a CF Function allowlist is a solid, well-layered setup.

The next limitation is not a security issue. It is a capability one: the photo list is static, and serving dynamic content requires a backend. Week 4 introduces EC2, ALB, and DynamoDB — and with a dynamic backend comes a new attack surface to secure.
