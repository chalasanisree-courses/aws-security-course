CloudFront and Edge Security

How Amazon CloudFront fixes the three Week 2 vulnerabilities — and the additional controls that make the edge a real security boundary.

Learning objectives
  • Explain what a CDN is and how CloudFront caches content at the edge
  • Describe what Origin Access Control does and why it's needed
  • Explain the difference between the S3 REST endpoint and the S3 website endpoint
  • Describe why bucket enumeration is structurally impossible through CloudFront
  • Explain how CloudFront Functions enforce an allowlist at the viewer request stage
  • Describe what CloudFront policies control and where each one sits in the request flow
  • Identify what Strict-Transport-Security does and when it matters

1. Problems with Week 2

Week 2 hosted the Photo Viewer directly from S3 static website hosting. It worked — but introduced three security and performance problems that make it unsuitable for a real application.

Issue 1 — HTTP only

The S3 website endpoint is HTTP only — TLS is not supported at this endpoint. All content travels in plaintext. Anyone on the same network can read it (man-in-the-middle). For a photo gallery this seems low risk — but the same pattern carries forward into later weeks where users will submit credentials and upload private photos.

Issue 2 — Public bucket

Static website hosting requires Block Public Access to be disabled and a policy granting Principal: *. Every file in the bucket is publicly accessible — not just web files. confidential.txt is readable by anyone who knows the path. And because s3:ListBucket was granted, anyone can enumerate the bucket with aws s3 ls --no-sign-request.

Issue 3 — No caching

The S3 bucket lives in us-east-1. Every request from every user — wherever they are — travels to Northern Virginia and back. For a real application serving users globally this latency is unacceptable and every request costs money.

2. The solution — CloudFront + edge security

We place CloudFront — Amazon's global CDN — in front of S3. This single change addresses all three Week 2 problems simultaneously, and adds several additional security controls at the edge.

3. Architecture

Week 3 inserts CloudFront between the browser and S3. The bucket goes fully private — only CloudFront can read it via OAC. Every request passes through CloudFront's edge before anything reaches S3.

5. Details

The table below shows which CloudFront control operates at each layer. This clarifies a common confusion — OAC and the CF Function are often mistaken for doing the same thing.

Click any row to explore each component in depth

☁️CloudFrontGlobal CDN · 400+ edge locations · HTTPS termination · cache · REST endpoint
›

CloudFront is a globally distributed cache. Requests are served from the nearest edge location — a user in Tokyo gets content from a Tokyo POP, not Northern Virginia. CloudFront also terminates HTTPS at the edge, replacing the HTTP-only S3 website endpoint entirely.

CloudFront reaches S3 via the REST endpoint — not the static website endpoint. The REST endpoint supports TLS and authenticated requests (required for OAC). The website endpoint supports neither.

Bucket enumeration is structurally impossible through CloudFront. CloudFront only issues GetObject requests for specific keys — there is no HTTP operation that maps to s3:ListBucket. The aws s3 ls --no-sign-request command that worked in Week 2 fails in Week 3 because it talks directly to the S3 API, which is now locked by OAC.

🔑Origin Access Control (OAC)SigV4 signing · bucket policy locks to your distribution ARN only
›

Adding CloudFront in front of S3 does not automatically make the bucket private — without OAC, anyone can bypass CloudFront and hit S3 directly. OAC uses two parts that must work together:

Part 1 — Request signing: CloudFront signs every request to S3 using AWS Signature Version 4. S3 verifies the signature proves it came from your specific distribution.

Part 2 — Bucket policy: Only allows s3:GetObject from requests signed by your distribution ARN. Everything else — direct S3 URL, CLI, any other tool — gets 403.

{
  "Principal": {"Service": "cloudfront.amazonaws.com"},
  "Action": "s3:GetObject",
  "Condition": {"StringEquals": {
    "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT:distribution/DIST_ID"
  }}
}

What OAC does not fix: OAC controls who can talk to S3. A request for /confidential.txt through CloudFront would still be served unless blocked by a CloudFront Function.

⚡CloudFront Function — allowlistViewer Request stage · blocks non-web files before cache · fails closed
›

Runs at the Viewer Request stage — before cache lookup or origin contact. Returns 403 immediately for blocked files, without ever touching S3.

function handler(event) {
    var uri = event.request.uri;
    var allowed = /\.(html|js|css|jpg)$/i;
    if (uri === '/' || allowed.test(uri)) return event.request;
    return { statusCode: 403, statusDescription: 'Forbidden' };
}

Allowlist vs blocklist: An allowlist permits only what is explicitly listed — everything else is blocked. A blocklist blocks only what is listed — everything else is permitted. Allowlists are more secure: a forgotten file type causes a blocklist to fail open, an allowlist to fail closed.

HSTS and login pages: For a static site HSTS has limited impact — no credentials to steal. But in Week 5 when we add a login page, HSTS becomes critical. It eliminates the window where a user typing http:// could have their password intercepted before the redirect to HTTPS. We add it now as infrastructure baseline.

🌏Geo-restrictionBlock or allow by country · no WAF required · distribution-level only
›

CloudFront has built-in geo-restriction — no WAF, no function, no extra cost. Two modes: allowlist (only listed countries can access) or blocklist (listed countries blocked). Blocked requests receive 403 at the nearest POP.

Limitation: applies to the entire distribution. For path-level geo-restriction (e.g. block one country from /admin only), use a CloudFront Function instead.

🔄Cache invalidationForce immediate edge refresh · always run after security changes
›

After updating content in S3 — patched JavaScript, new CF Function code, updated policies — edge caches may still serve the old version until TTL expires. Create an invalidation to purge immediately: use /* for everything or specific paths like /app.js for targeted invalidation.

Always invalidate after security-relevant changes. Without invalidation, stale cached responses — missing your new headers, function behavior, or patched code — continue to be served globally until TTL expires.

6. Problems with this solution

Week 3 fixes all three Week 2 vulnerabilities. But it introduces a new one — and leaves one issue unresolved that will matter more in future weeks.

Static photo list — no dynamic content yet
click to expand

The limitation: Photos are still listed in a static photos.json file. Adding a new photo requires editing the JSON, re-uploading it, and creating a CloudFront cache invalidation. There is no way to add content at runtime. The app has no backend — everything is static. Fixed in Week 4 by introducing EC2, DynamoDB, and a dynamic API endpoint.

No WAF — no rate limiting or advanced threat protection
click to expand

The limitation: CloudFront provides Shield Standard (L3/L4 DDoS protection) automatically and for free. But there is no Web Application Firewall — no rate limiting, no SQL injection protection, no bot detection. Any IP can make unlimited requests. Fixed in Week 9 by adding AWS WAF to the CloudFront distribution.
