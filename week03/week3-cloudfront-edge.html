CloudFront and Edge Security

How Amazon CloudFront fixes the three Week 2 vulnerabilities — and the additional controls that make the edge a real security boundary.

Learning objectives

Issue 1 — HTTP only

The S3 static website endpoint (http://bucket.s3-website-us-east-1.amazonaws.com) is HTTP only. There is no TLS support at this endpoint — you cannot enable HTTPS on it. All content is served in plaintext, which means anyone on the same network can read the data in transit. This is called a man-in-the-middle attack. For a purely public photo gallery it may seem low-risk, but the same pattern carries forward into later stages where users will submit credentials and upload private photos.

Issue 2 — Public bucket

For S3 static website hosting to work, Block Public Access must be disabled and a bucket policy must grant s3:GetObject and s3:ListBucket to Principal: *. That means every file in the bucket — not just the web files — is publicly accessible. In Week 2 we deliberately placed confidential.txt in the bucket to demonstrate this. Anyone who knows the path can read it. And because s3:ListBucket was granted, anyone can enumerate the bucket with aws s3 ls --no-sign-request to discover files they didn't know existed.

Issue 3 — No caching

The S3 bucket lives in us-east-1 (Northern Virginia). Every request — from every user, from every location — travels all the way to us-east-1 and back. For a small static site with tiny image files this is tolerable. But for any real application serving users globally, this round-trip latency is unacceptable. There is no edge caching, no regional serving, no way to bring content closer to users.

2. What CloudFront is

CloudFront is Amazon's content delivery network — a globally distributed cache with over 400 edge locations worldwide. When you put CloudFront in front of an origin (in our case, S3), requests are served from the edge location nearest to the user rather than always hitting the origin.

REST endpoint vs website endpoint

CloudFront talks to S3 using the REST endpoint, not the static website endpoint used in Week 2.

Endpoint type Format TLS support Used for
Website endpoint bucket.s3-website-region.amazonaws.com No — HTTP only Week 2 static hosting. Requires public access.
REST endpoint bucket.s3.region.amazonaws.com Yes — TLS always CloudFront + OAC. Works with private buckets.

Why this matters for OAC: Origin Access Control signs requests using AWS Signature Version 4. SigV4 only works against the REST API — the website endpoint does not support authenticated requests. This is why when you create a CloudFront distribution with S3, the console shows two entries for your bucket and you must pick the REST endpoint, not the website URL.

Bucket enumeration is structurally impossible through CloudFront

Users accessing the Photo Viewer through CloudFront can never enumerate the S3 bucket — not because of a policy restriction, but because of how CloudFront works. CloudFront only issues GetObject requests for specific keys. There is no HTTP operation that translates to s3:ListBucket. The S3 API operation that lists bucket contents cannot be triggered through an HTTP request to CloudFront — it simply doesn't exist at that layer.

This means aws s3 ls --no-sign-request — which worked in Week 2 — fails in Week 3 regardless of bucket policy, because that tool talks directly to the S3 API, which is now locked by OAC.

3. Origin Access Control (OAC)

Adding CloudFront in front of S3 does not automatically make S3 private. If the bucket is still public, users can bypass CloudFront entirely and hit the S3 REST endpoint directly. OAC solves this.

OAC — Origin Access Control — is a CloudFront feature that does two things simultaneously:

Part 1 — Request signing

CloudFront signs every request it sends to S3 using AWS Signature Version 4. S3 can verify the signature proves the request is genuinely from your specific CloudFront distribution — not someone else faking it.

Part 2 — Bucket policy update

The S3 bucket policy is updated to only allow s3:GetObject requests where the signature matches your specific CloudFront distribution ARN. Every other request — direct S3 URL, CLI, any other tool — gets 403.

Neither part works without the other. Signing without the restrictive bucket policy means S3 still accepts unsigned requests. A restrictive bucket policy without signing means CloudFront can't read anything. Both must be configured together.

In the new AWS console: When you check "Allow private S3 bucket access to CloudFront — Recommended" during distribution creation, AWS creates the OAC and writes the bucket policy automatically. In the old console, these were separate manual steps.

The OAC bucket policy

The policy CloudFront generates looks like this. The key line is the Condition — it restricts access to one specific CloudFront distribution by ARN, not just any CloudFront distribution.

{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "cloudfront.amazonaws.com"
    },
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::your-bucket/*",
    "Condition": {
      "StringEquals": {
        "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT_ID:distribution/DIST_ID"
      }
    }
  }]
}

What OAC does not fix

OAC controls who can talk to S3 — it does not control what CloudFront will serve. If a user knows the path /confidential.txt and requests it through CloudFront, CloudFront will fetch it from S3 and serve it. OAC fixed the direct S3 access attack. It did not fix the path-based access attack through CloudFront. That requires a CloudFront Function.

4. CloudFront Functions

A CloudFront Function is a small piece of JavaScript that runs at the edge at the viewer request stage — before CloudFront checks its cache or contacts the origin. It can inspect the incoming request and return a response immediately, without the request ever reaching S3.

The allowlist function

For the Photo Viewer we write an allowlist function — it defines exactly which file types are permitted. Anything not on the list returns 403 at the edge.

function handler(event) {
    var uri = event.request.uri;

    // Allow the root path — CloudFront serves index.html via default root object
    // Allow only .html, .js, .css, and .jpg files — regardless of folder depth
    var allowed = /\.(html|js|css|jpg)$/i;

    if (uri === '/' || allowed.test(uri)) {
        return event.request;
    }

    // Everything else — .txt, .env, .zip, any unknown extension — is blocked
    return {
        statusCode: 403,
        statusDescription: 'Forbidden'
    };
}

Allowlist vs blocklist: An allowlist defines exactly what is permitted and blocks everything else. A blocklist defines what is blocked and permits everything else. Allowlists are more secure because they don't require you to anticipate every possible attack. If you forget to block one file type, a blocklist fails open — an allowlist fails closed.

Where it runs: The function is attached to the distribution behavior as a Viewer Request function. This means it runs at the edge, before cache lookup. A request for confidential.txt is blocked at the nearest CloudFront POP — it never reaches the cache, never reaches S3.

5. CloudFront policies

CloudFront has three types of policies that attach to different parts of the request-response flow. Understanding where each one sits explains what it can and cannot control.

Click each policy to expand

Viewer Viewer Protocol Policy
Controls browser ↔ CloudFront connection ▼

Tells CloudFront how to handle the connection between the browser and the edge. Three options: allow both HTTP and HTTPS, redirect HTTP to HTTPS automatically, or require HTTPS only and reject HTTP entirely.

For the Photo Viewer we use Redirect HTTP to HTTPS. If a user types http://, CloudFront catches the request at the edge and returns a 301 redirect to the HTTPS version. The browser then re-requests over HTTPS. The redirect itself is unencrypted — that brief window is addressed by Strict-Transport-Security (see Response Headers Policy below).

This is a native CloudFront setting, not a function or WAF rule — it requires no code.

Cache Cache Policy
Controls what CloudFront stores and for how long ▼

Controls what CloudFront stores at the edge and for how long — reducing origin requests and improving performance. The key setting is TTL (Time To Live) — how long a cached object is considered fresh before CloudFront goes back to S3 for a new copy.

Security relevance: TTL affects how quickly a security fix propagates. If you patch a vulnerability in a file and the TTL is 24 hours, the old cached version is still being served globally for up to 24 hours. Use invalidations (/*) to force immediate cache clearing after security-relevant changes.

For the Photo Viewer we use the AWS managed CachingOptimized policy, which is the recommended default for S3 origins.

Origin Origin Request Policy
Controls what CloudFront forwards to S3 ▼

When CloudFront has a cache miss and needs to fetch content from S3, it constructs an origin request. The Origin Request Policy controls exactly which headers, cookies, and query strings from the user's request are included in that origin request.

Security relevance: Query strings and headers can carry malicious payloads. An attacker can craft a URL with injected code in the query string, hoping it reaches a vulnerable backend. By forwarding only what the origin actually needs — and stripping everything else — you reduce the attack surface. This is the least-privilege principle applied to HTTP request forwarding.

For Week 3: S3 ignores query strings, so the security benefit is limited right now. This becomes important in Week 5 when Lambda becomes the origin — those services do process query strings and headers.

Response Response Headers Policy
Adds security headers to every browser response ▼

Adds HTTP security headers to every response CloudFront sends to the browser — improving security posture without touching application code. AWS provides a managed SecurityHeadersPolicy that includes the most important headers.

Header What it does Defends against
Strict-Transport-Security After first visit, browser always upgrades to HTTPS internally — never sends an HTTP request Man-in-the-middle on the HTTP→HTTPS redirect window
X-Frame-Options Prevents the page from being embedded in an <iframe> on another domain Clickjacking — tricking users into clicking invisible UI elements
X-Content-Type-Options Tells the browser not to guess the file type — trust the Content-Type header exactly MIME sniffing — browser executing a .jpg that contains JavaScript

Strict-Transport-Security and login pages: For a purely static site, HSTS has limited practical impact — there are no credentials to steal. But in Week 5 when we add a login page, HSTS becomes critical. A user on a coffee-shop WiFi who types http:// could have their username and password intercepted in that brief HTTP window before CloudFront redirects them. HSTS eliminates that window by making the browser skip HTTP entirely after the first visit. We add it now as infrastructure baseline.

6. Where each control sits in the request flow

The diagram below shows which CloudFront control applies at each stage of a request. Understanding this prevents a common confusion — students often assume OAC and the CF function do the same thing.

Layer What runs here What it controls
Viewer Request CF Function (allowlist) Blocks non-web file types before cache or origin. confidential.txt → 403 here.
Protocol Viewer Protocol Policy HTTP → HTTPS redirect. Happens before any content is served.
Cache Cache Policy TTL, cache key. Cache hit → serve immediately, no origin request.
Origin Request OAC + Origin Request Policy OAC signs request with CloudFront identity. Origin Request Policy strips unnecessary headers/cookies.
S3 bucket OAC bucket policy Only allows requests signed by your specific CloudFront distribution. All other callers → 403.
Viewer Response Response Headers Policy Adds HSTS, X-Frame-Options, X-Content-Type-Options to every response.

8. Geo-restriction

CloudFront has a built-in geo-restriction feature that blocks or allows requests based on the geographic location of the user's IP address. This is a native CloudFront setting — no WAF, no function, no extra cost.

Two modes: allowlist (only listed countries can access) and blocklist (listed countries are blocked, everyone else can access). Blocked requests receive a 403 response at the nearest edge location — they never reach your origin.

The control sits in the distribution's Security tab → CloudFront geographic restrictions. It applies to the entire distribution — you cannot restrict one path differently from another using this feature alone. For path-level geo-restriction, a CloudFront Function is needed.

9. Cache invalidation

When you update content in S3 — new photos, updated CSS, patched JavaScript — CloudFront edge caches may still serve the old version until the TTL expires. To force immediate refresh, you create an invalidation.

An invalidation tells CloudFront to purge matching objects from all edge caches globally. Use /* to invalidate everything, or specify individual paths like /app.js for targeted invalidation.

Always invalidate after security changes. If you update a CloudFront Function, attach a new policy, or patch a file in S3, create a /* invalidation immediately. Otherwise stale cached content continues to be served — including any previously cached responses that don't include your new security headers or function behavior.
