Week 3 Lab — CloudFront + Edge Security

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  100 pts

Objective: Fix all three Week 2 vulnerabilities by placing CloudFront in front of S3. The bucket goes fully private. HTTPS is enforced. A CloudFront Function blocks non-web files at the edge. Security response headers are added to every response.

Files for this lab: Get all files from the course GitHub repository → photoviewer/week03

Part 1 — Lock down the S3 bucket

Step 1Upload updated files to S3

Go to your bucket root → Upload → add the updated app.js and photos.json from the Week 3 GitHub folder → Upload.

When prompted about overwriting the existing app.js — confirm the overwrite.

Step 2Disable static website hosting

Go to your bucket → Properties tab → Static website hosting → Edit:

  • Static website hosting: Disable → Save changes
Step 3Enable Block Public Access and remove bucket policy

Go to your bucket → Permissions tab → Block public access → Edit:

  • Check all four boxes → Save changes → type confirm → Confirm

Still on Permissions tab → Bucket policy → Edit → delete all content → Save changes.

Screenshot: Permissions tab showing Block Public Access fully ON and empty bucket policy
Step 4Confirm the old S3 website is now blocked

Try to access your old S3 website endpoint in the browser — it should now return 403 Forbidden or be completely inaccessible.

The URL looks like: http://photoviewer-[your-account-id].s3-website-us-east-1.amazonaws.com

Screenshot: Browser showing 403 or access denied on the old S3 website URL — make sure the URL is visible in the screenshot

Part 2 — Create CloudFront distribution

Step 5Create the distribution with OAC

Go to CloudFront → Distributions → Create distribution:

Origin settings

  • Origin domain: select your S3 bucket from the dropdown — use the REST endpoint (e.g. photoviewer-xxx.s3.amazonaws.com), not the website endpoint
  • When prompted "Allow private S3 bucket access to CloudFront?" → click Yes, update the bucket policy — this creates OAC and updates the bucket policy automatically

Default cache behavior

  • Viewer protocol policy: Redirect HTTP to HTTPS
  • Cache policy: CachingOptimized (recommended for S3)
  • Response headers policy: select SecurityHeadersPolicy (AWS managed)
If you don't see the Response headers policy option during creation, don't worry — you can set it afterward by going to Behaviors tab → Edit default behavior → Response headers policy.

Settings

  • Default root object: index.html — don't skip this or your CloudFront URL will return an error instead of the Photo Viewer
  • WAF: leave disabled

Click Create distribution — wait for Status: Enabled (~2 minutes).

Screenshot: CloudFront distribution Details page showing Enabled status and your CloudFront domain name — URL must be visible
Screenshot: S3 bucket Permissions tab showing the OAC bucket policy that CloudFront automatically created

Part 3 — Verify basics

Step 6Verify HTTPS and HTTP redirect

✅ Photo Viewer loads over HTTPS

Open your CloudFront domain in the browser. The Photo Viewer should load with a padlock visible.

Screenshot: Photo Viewer loading at the CloudFront URL over HTTPS — padlock and URL must be visible

✅ HTTP redirects to HTTPS

Type http:// before your CloudFront domain and press Enter. Confirm it automatically redirects to HTTPS.

Screenshot: Browser showing the redirect from HTTP to HTTPS — URL must be visible
Step 7Verify bucket enumeration is blocked

In CloudShell, run:

aws s3 ls s3://photoviewer-[your-account-id] --no-sign-request

Should return Access Denied — the bucket is now private.

Screenshot: CloudShell showing Access Denied on --no-sign-request
Step 8Check confidential.txt — before the CF Function

Navigate to https://[your-cf-domain]/confidential.txt in your browser.

Think: Is it accessible or blocked? OAC ensures only CloudFront can access S3 — but does that mean CloudFront blocks specific files? In one sentence, explain what you see and why.
Screenshot: Browser response for /confidential.txt through CloudFront — URL must be visible. Include your one-sentence explanation in your submission.

Part 4 — CloudFront Function

Step 9Create the allowlist function

Go to CloudFront → Functions → Create function:

  • Name: photoviewer-allowlist
  • Runtime: cloudfront-js-2.0
  • Paste the code from photoviewer-allowlist.js in the GitHub repo
  • Click Save changes → Publish — publishing is required before you can attach the function
Important: You must click Publish after saving. An unpublished function cannot be attached to a distribution.
Step 10Attach the function and invalidate

Go to your distribution → Behaviors tab → select the default behavior → Edit:

  • Function associations → Viewer request → select photoviewer-allowlist
  • Click Save changes

Go to Invalidations tab → Create invalidation → path: /* → Create invalidation. Wait for status: Completed.

Screenshot: Default behavior showing photoviewer-allowlist attached at Viewer Request
Step 11Verify the function works correctly

✅ Non-web files blocked

Navigate to https://[your-cf-domain]/confidential.txt again — should now return 403 Forbidden.

Screenshot: 403 response on /confidential.txt through CloudFront — URL must be visible

✅ Legitimate content still loads

Reload the Photo Viewer at your CloudFront URL — photos should still display correctly. The allowlist must not be blocking legitimate content.

Screenshot: Photo Viewer loading photos correctly after the CF Function is attached

Part 5 — Security headers

Step 12Verify SecurityHeadersPolicy is attached and review its contents

If you attached SecurityHeadersPolicy during distribution creation, it is already active. If not, go to Behaviors tab → Edit default behavior → Response headers policy → SecurityHeadersPolicy → Save changes.

View the policy details

Go to CloudFront → Policies → Response headers → find SecurityHeadersPolicy → click to view details.

Screenshot: SecurityHeadersPolicy details page showing the security headers it configures

Verify headers in browser

Open DevTools → Network tab → click any request → Response Headers. Confirm all three are present:

  • Strict-Transport-Security
  • X-Frame-Options
  • X-Content-Type-Options
Screenshot: DevTools Response Headers showing all three security headers

Reflection

Answer both questions in 2–3 sentences each.

Question 1

Origin Access Control (OAC) ensures only CloudFront can access your S3 bucket — yet in Step 8 you saw that confidential.txt was still accessible through CloudFront before the CF Function was added. Explain why OAC alone is not enough to block specific files, and what problem OAC actually solves.

Question 2

The CloudFront Function uses an allowlist rather than a blocklist to control which file types are served. Explain the difference and why an allowlist is more secure. What would happen if you forgot to add a file type to a blocklist vs an allowlist?
