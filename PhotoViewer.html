Photo Viewer

A cloud-native web application built and secured from scratch — week by week.
Each week adds new AWS services, new functionality, and new security challenges to solve.

Weekly application builds
Week 2
Static S3 Hosting
Lab complete

A static HTML/CSS/JavaScript photo viewer hosted directly from a public S3 bucket. The photo list is stored in a static photos.json file. No backend, no database, no HTTPS.

Traffic Browser → S3 (HTTP)
Photo list photos.json (static)
Photos S3 (public bucket)
Deliberate vulnerabilities
HTTP only — no TLS, content interceptable in transit
Public bucket — confidential.txt readable by anyone
Bucket enumerable — --no-sign-request lists all files
No caching — every request goes to us-east-1
⌥ View code on GitHub
Week 3
CloudFront + Edge Security
Lab complete

CloudFront placed in front of S3 with Origin Access Control. S3 bucket made fully private. HTTPS enforced at the edge. A CloudFront Function allowlist blocks non-web file types. Security response headers added.

Traffic Browser → CloudFront (HTTPS) → S3 (OAC)
Photo list photos.json (static)
Edge CF Function (allowlist)
Fixed from Week 2
HTTPS enforced — HTTP redirected automatically
S3 private — OAC restricts access to CloudFront only
Non-web files blocked at edge — returns 403
Global edge caching — 400+ CloudFront POPs
⌥ View code on GitHub
Week 4
EC2 Backend in VPC
Lab complete

Photo metadata moved from a static JSON file to DynamoDB. An application server on EC2 in a private VPC subnet serves the photo list dynamically. CloudFront routes /photos API requests to an ALB which forwards to EC2.

Traffic Browser → CloudFront → ALB → EC2
Photo list DynamoDB (dynamic)
Access SSM — no SSH
Remaining vulnerabilities
ALB internet-facing — CloudFront bypassable via ALB DNS
No rate limiting on /photos API — DDoS risk
No authentication — photo list publicly accessible
⌥ View code on GitHub
Full application roadmap
Week 2
Static S3 Hosting
Static HTML/CSS/JS hosted on S3. Photo list stored in a static JSON file. Deliberately insecure — HTTP only, public bucket, no caching.
Week 3
CloudFront + Edge Security
CloudFront with OAC privatizes S3. HTTPS enforced. CF Function blocks non-web files at the edge. Response headers policy adds HSTS, X-Frame-Options, X-Content-Type-Options.
Week 4
EC2 Backend in VPC
Application server on EC2 in a private subnet serves photo list from DynamoDB. ALB in public subnet. VPC endpoints for DynamoDB, S3, and SSM. No SSH — SSM Session Manager only.
Week 5
Serverless Migration coming
EC2 replaced with Lambda + API Gateway. Same DynamoDB table, same API response. Eliminates the entire EC2 attack surface — no patching, no SSM, no VPC endpoints needed.
Week 6
Authentication
OIDC login via Cognito. Free vs premium user tiers. Private photos visible only to their owner. HttpOnly/Secure/SameSite cookies.
Week 7
IAM Identity Center
Centralized access management for the application team. Role-based access — developers, admins, auditors. Replaces individual IAM users.
Week 8+
Hardening — WAF, KMS, Macie, GuardDuty
WAF rate limiting and SQLi/XSS rules. KMS encryption for DynamoDB and S3. Macie PII detection on uploaded photos. GuardDuty threat detection. Security Hub posture dashboard.
