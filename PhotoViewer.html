CS 55D — Course Project

Photo Viewer

A cloud-native web application built and secured from scratch — week by week.
Each stage adds new functionality and introduces new security challenges to solve.

Application stages
Stage 1
Static S3 site
Week 2 — Lab complete

A static HTML/CSS/JavaScript photo viewer hosted directly from an S3 bucket with static website hosting enabled. Five photos served from a hardcoded list in a JSON file. No backend, no database, no authentication.

Traffic Browser → S3 website endpoint
Storage S3 (public bucket)
Photos photos.json (static)
Deliberate vulnerabilities
HTTP only — no TLS, content interceptable in transit
Public bucket — confidential.txt accessible to anyone
Bucket enumerable — --no-sign-request lists all files
No caching — every request travels to us-east-1
Stage 2
CloudFront + Edge Security
Week 3 — Lab complete

CloudFront placed in front of S3 with Origin Access Control. S3 bucket made fully private — only CloudFront can read it. HTTPS enforced. A CloudFront Function blocks non-web file types at the edge. Security response headers added.

Traffic Browser → CloudFront (HTTPS) → S3 (OAC)
Storage S3 (private bucket)
Edge CF Function (allowlist)
What was fixed
HTTPS enforced — HTTP redirected automatically
S3 bucket private — OAC restricts access to CloudFront only
Non-web files blocked — CF Function allowlist returns 403
Global edge caching — 400+ CloudFront POPs worldwide
Full application roadmap
Week 2
Stage 1 — Static S3 site
Static HTML/CSS/JS hosted on S3. Five photos from a static JSON list. Deliberately insecure — HTTP only, public bucket, no caching.
Week 3
Stage 2 — CloudFront + Edge Security
CloudFront with OAC privatizes S3. HTTPS enforced. CF Function blocks non-web files. Response headers policy adds HSTS, X-Frame-Options, X-Content-Type-Options.
Week 4
Stage 3 — EC2 backend in VPC coming
EC2 Flask server in a private subnet serves the photo list dynamically from DynamoDB. ALB in public subnet. CloudFront routes /photos API calls to ALB. VPC, security groups, SSM, Inspector, CloudWatch.
Week 5
Stage 4 — Serverless migration
EC2 backend replaced with Lambda + API Gateway. Same DynamoDB table, same API response. Demonstrate why serverless eliminates entire EC2 attack surface.
Week 6
Stage 5 — Authentication
OIDC login via Cognito. Free vs premium user tiers. Server-side sessions in DynamoDB. HttpOnly/Secure/SameSite cookies.
Week 7+
Stages 6–8 — Hardening
IAM Identity Center, WAF, KMS encryption, Macie PII detection, Rekognition content safety, GuardDuty threat detection, Security Hub posture dashboard.
