Application build stages
Week 1
AWS account baseline
MFA on root, IAM admin user, CloudTrail, billing alarm, Cost Anomaly Detection — no application yet
Week 2
Photo browsing — public, unauthenticated, no login required
Any visitor can click through photos. No account needed.
›
What users can do
Any visitor can open the Photo Viewer and click through a set of travel photos — no login, no account. The photo list is fixed at five photos. There is no search, no upload, no user accounts.
How it is built on AWS
Browser → HTTP → S3 static website hosting
HTML, CSS, JavaScript, and a static photos.json file all live in a single public S3 bucket. The browser fetches photos.json to get the photo list, then requests each image. S3 serves everything. There is no backend server — all logic runs in the browser.
Limitations of this solution
Security Functionality Performance Operability
  • HTTP only — no TLS. Traffic between the browser and S3 is unencrypted. A man-in-the-middle can read or inject content into any response.
  • Public bucket — the entire bucket is publicly readable. Any file, including confidential.txt, is accessible to anyone who knows the URL. Bucket contents are enumerable.
  • No edge caching — every request travels all the way to us-east-1 regardless of where the user is. Slow for users outside the US.
  • Photo list is static — adding a photo means manually editing photos.json and redeploying. Not scalable.

Week 3
Photo browsing — now secure and globally fast
Same unauthenticated experience. HTTPS enforced. S3 bucket privatized. Content delivered from edge locations worldwide.
›
What users can do
The experience is identical to Week 2 — visitors click through photos without logging in. The improvements are entirely infrastructure: the app is now served over HTTPS, loads faster globally, and the S3 bucket is no longer publicly accessible.
How it is built on AWS
Browser → HTTPS → CloudFront → OAC → S3 (private bucket)
CloudFront sits in front of S3. Origin Access Control (OAC) makes the S3 bucket fully private — only CloudFront can read from it. HTTPS is enforced at the edge; HTTP requests are automatically redirected. A CloudFront Function blocks non-web file types at the viewer request stage. Response headers add HSTS, X-Frame-Options, and X-Content-Type-Options.
Limitations of this solution
Security Functionality Performance Operability
  • Photo list is still static — photos.json is still a file deployed with the application. Adding or removing photos requires a redeployment. A real photo-sharing platform needs a database.

Week 4
Photo catalog becomes dynamic — photos managed in a database
Still unauthenticated browsing. Photo list now served from DynamoDB — no redeployment needed to add photos.
›
What users can do
Visitors still browse photos without logging in — the user-facing experience is unchanged. The difference is architectural: the photo catalog is now stored in DynamoDB. Add a row to the database and it appears to all visitors immediately, with no redeployment.
How it is built on AWS
Browser → HTTPS → CloudFront → ALB (public subnet) → EC2 app server (private subnet) → DynamoDB
CloudFront now has two origins: S3 for static files (HTML, CSS, JS, images) and an ALB for the /photos API endpoint. The ALB forwards /photos requests to an EC2 application server running in a private VPC subnet — no public IP, no SSH. SSM Session Manager handles all remote access. The app server queries DynamoDB via a VPC Gateway endpoint and returns the photo list as JSON.
Limitations of this solution
Security Functionality Performance Operability
  • ALB is internet-facing — its public DNS name is reachable directly, bypassing CloudFront and all edge security controls.
  • Supply chain risk — the EC2 bootstrap script installs packages from the public internet. A compromised package would run as root, inherit the instance role, and could exfiltrate data.
  • No rate limiting on /photos — the API endpoint has no throttling. A flood of requests goes straight through CloudFront to the ALB and EC2, consuming resources with no protection.
  • EC2 requires ongoing maintenance — OS patching, SSM agent updates, capacity management. Every hour the instance runs costs money even during idle periods.
  • No user accounts — the platform still cannot distinguish visitors, offer a free tier, or support premium features.

Week 5
Serverless backend — same experience, better architecture
EC2 and ALB replaced with Lambda and API Gateway. No servers to patch or manage.
›
What users can do
The user-facing experience is identical to Week 4 — unauthenticated visitors browse photos served from DynamoDB. The change is entirely architectural: the backend is now serverless.
How it is built on AWS
Browser → HTTPS → CloudFront → API Gateway → Lambda → DynamoDB
EC2 and ALB are replaced by Lambda and API Gateway. The same DynamoDB table and CloudFront distribution from Week 4 are reused unchanged. API Gateway handles HTTPS natively — no certificate management needed. Lambda runs only when a request arrives — no idle cost, no OS to patch, no SSH surface.
