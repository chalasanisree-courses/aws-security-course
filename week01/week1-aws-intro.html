1. What is Cloud Computing?

Before AWS existed, if a company wanted to run a web application, they had to buy servers — physical machines — rack them in a data center, wire up networking, install an operating system, and manage all of it forever. Capacity planning was a guessing game: buy too little and your app crashes under load; buy too much and the hardware sits idle, costing money.

Cloud computing changes the economic model. Instead of buying hardware, you rent computing resources on demand — paying only for what you use, scaling up or down in minutes, and handing the physical infrastructure problem to someone else entirely.

AWS is the world's largest cloud provider. It owns and operates data centers in dozens of countries and rents access to the compute, storage, networking, and managed services inside them. When you launch an EC2 instance, you are renting a slice of a server in an Amazon data center — you never see the hardware, never rack it, never replace a failed disk.

Traditional IT (on-premises)

⏳Weeks or months to provision new capacity
💰Large upfront capital expense — buy before you know the demand
🔧Your team manages hardware, power, cooling, cabling
📦Idle capacity during off-peak hours — still paying for it
🌍Expanding to a new country means building or leasing a data center

Cloud computing (AWS)

⚡Provision a new server in seconds via console or API
💳Pay-as-you-go — operational expense, not capital expense
🏗AWS manages physical infrastructure — you manage configuration
📈Scale to zero when idle, scale to thousands under load
🌍Deploy to a new region in minutes — AWS already has the data centers
The security implication: Moving to cloud shifts which security problems you have, not whether you have security problems. Physical server theft is no longer your concern — but misconfigured access policies, exposed S3 buckets, and over-permissioned IAM roles are. The threat model changes; the need for security does not.

2. Cloud service models — IaaS, PaaS, SaaS

Cloud providers don't all offer the same thing. Think of it as a spectrum from "we hand you a raw machine, you figure out the rest" all the way to "just tell us what you want — we handle everything underneath." These three models describe where on that spectrum a given service sits.

The model matters for security: whatever the provider manages, you can't misconfigure. Whatever you manage, you can. More control means more responsibility.

Click each model to expand it.

IaaS

Infrastructure as a Service

You manage the OS and everything above

PaaS

Platform as a Service

You manage the code and config

SaaS

Software as a Service

You manage your data and access

The provider gives you virtual hardware — a machine, a disk, a network interface. You install the OS, you patch it, you run your software on it.

What the provider handles: physical servers, networking equipment, power, cooling, and the hypervisor that runs your virtual machine.

What you handle: operating system installation and patching, firewall rules, application software, and all security configuration above the hypervisor.

Analogy: Baking a birthday cake from scratch. You buy the raw ingredients, mix the batter, bake it, and decorate it yourself. Total control — but every step is your responsibility. If it burns, that's on you.

Examples: AWS EC2 Azure Virtual Machines Google Compute Engine

The provider manages the OS, runtime, and scaling. You bring your application code — the platform handles everything underneath it.

What the provider handles: OS patching, runtime updates, scaling infrastructure, and availability. You never touch a server.

What you handle: your application code, access permissions, encryption settings, and environment configuration.

Analogy: Buying a plain pre-baked cake and decorating it yourself. The hard part is done. You focus on the frosting and decorations — you can still make it look terrible, but you didn't have to worry about whether it rose properly.

Examples: AWS Lambda / RDS Azure App Service Google App Engine Heroku

The provider runs the entire application. You log in and use it. There is no infrastructure or code for you to manage at all.

What the provider handles: everything — servers, OS, application code, uptime, patching, scaling.

What you handle: who in your organization has access, what data you put in, and how you configure the application for your needs.

Analogy: Ordering a fully decorated birthday cake from a bakery. You just show up and enjoy it. Your only job is deciding who gets a slice and not leaving it out in the rain.

Examples: Gmail / Google Workspace Salesforce Microsoft 365 Slack

3. AWS and its services

AWS (Amazon Web Services) is the world's largest cloud provider, launched in 2006. It holds roughly a third of the global cloud market — ahead of Microsoft Azure and Google Cloud. Most large enterprises and a majority of internet startups run on AWS, which is why it's the focus of this course.

AWS offers over 200 services spanning all three service models. The services you'll encounter most in this course:

IaaS Infrastructure
EC2Virtual machines — rent a server by the hour
VPCPrivate network — your isolated slice of AWS networking
EBSPersistent disk storage attached to EC2
PaaS Managed compute & data
LambdaRun code without managing servers — pay per invocation
RDSManaged relational database — AWS patches the DB engine
API GatewayManaged HTTP API layer — routes requests to your backend
SaaS / Managed Fully managed services
S3Object storage — store any file, any size, durably
DynamoDBManaged NoSQL database — no servers, scales automatically

IAMIdentity & access management — controls who can do what
Why AWS over Azure or Google Cloud? All three major providers offer equivalent capabilities. AWS has the largest market share, the most mature security tooling, and the most employer demand — which is why it's the industry standard starting point for cloud security certifications and roles.

4. AWS Shared Responsibility Model

When you move to the cloud, you don't hand security over to AWS — you share it. AWS formalizes this as the Shared Responsibility Model, summarized in one line:

The core principle

AWS is responsible for security of the cloud. You are responsible for security in the cloud.

AWS secures the physical infrastructure — the data centers, the hardware, the hypervisors, the global network. That layer is entirely AWS's problem. What you run on top of that infrastructure — your operating systems, your applications, your data, your access controls — is your problem.

The diagram below shows how this split works. Think of the stack as layers: AWS owns everything at the bottom; you own everything at the top. Where the line sits depends on which service model you use — the more managed the service, the higher AWS's responsibility reaches.

Notice the pattern: as you move from IaaS to SaaS, the responsibility boundary moves up the stack. With IaaS you own everything from the OS upward. With SaaS you own almost nothing — but you are still always responsible for who has access and what data you put in. Those never transfer to the provider.

The most important implication: The vast majority of cloud security incidents are not caused by AWS infrastructure failures. They are caused by customer misconfiguration — an open S3 bucket, an overly permissive IAM policy, an unrotated credential. The Shared Responsibility Model makes clear that these are your responsibility, not AWS's.

5. AWS Well-Architected Framework

The Well-Architected Framework is AWS's codified set of design principles for building cloud systems. It emerged from AWS solutions architects reviewing thousands of customer architectures and distilling what consistently led to good outcomes — and what consistently led to outages, breaches, and runaway costs.

It has six pillars. Security is the focus of this course, but the pillars are not independent — an unreliable system that crashes is not truly secure; a system you can't monitor can't be audited. Click any pillar to learn more.

🔐

Security

Protect data, systems, assets

Covers identity foundation (least privilege, no long-term credentials), traceability (audit logs for all actions), layered defense (edge, network, compute, application, data), data protection in transit and at rest, and incident preparedness.
Course focus
⚙️

Operational Excellence

Run, monitor, improve

Running workloads effectively and gaining operational insight. Key practices: infrastructure as code, automated responses, runbooks, and continuous improvement. A system that can't be observed can't be secured.
🔄

Reliability

Recover from failures

The ability to perform intended functions correctly and recover from failures. Availability zones and multi-region deployments are the primary mechanisms. Reliability and security overlap — both require redundancy and failure planning.
⚡

Performance Efficiency

Use resources efficiently

Using computing resources efficiently at any scale. Choosing the right service type for the job — managed services often outperform self-managed infrastructure at scale while reducing the operational burden.
💰

Cost Optimization

Avoid unnecessary costs

Avoiding unnecessary costs by understanding spending, right-sizing resources, and eliminating waste. In cloud environments, unused resources still cost money — cost discipline is part of good architecture.
🌱

Sustainability

Minimize environmental impact

Added as the sixth pillar in 2022. Covers selecting efficient regions, right-sizing resources to avoid waste, and preferring managed services over self-managed infrastructure. Cloud providers can optimize energy use across many customers more effectively than individual organizations can on-premises.
Well-Architected and Shared Responsibility connect: The Well-Architected Framework tells you how to build well. The Shared Responsibility Model tells you what you own and must secure. Together they define the standard you're working toward and the boundary of what you're accountable for.
