Week 1 Lab — AWS Account Setup

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  25 pts

Objective: Set up your AWS account security baseline. You will complete some steps as root, then create an IAM admin user and switch to that user for the rest of the course. Root is used only where required — never for day-to-day work.

Important: Use your own personal AWS account — not a shared or employer account. You need full admin access and will be creating and deleting resources throughout this course. AWS Free Tier is sufficient.
🔴 Do the following steps as ROOT
Step 1Create your AWS account

Go to aws.amazon.com/free and sign up for a free account. Use an email address only you have access to — this becomes your root account email.

If you already have an AWS account, sign in as root (your account email address).

Make a note of your 12-digit account ID — visible in the top-right corner of the console. You will need it throughout this course.
Step 2Enable MFA on root

Go to the top-right menu → Security credentials:

  • Scroll to Multi-factor authentication (MFA) → Assign MFA device
  • Device name: root-mfa
  • MFA device type: Authenticator app (Google Authenticator, Authy, or similar)
  • Scan the QR code → enter two consecutive codes → Add MFA
Screenshot: Security credentials page showing MFA device assigned to root
Step 3Enable CloudWatch billing alerts

Go to Billing and Cost Management → Billing preferences:

  • Check Receive CloudWatch billing alerts → Save preferences
This allows CloudWatch to receive billing events so you can create alarms. It must be enabled by root — an IAM user cannot do this.
Step 4Create a budget alarm

Go to Billing and Cost Management → Budgets → Create budget:

  • Budget type: Cost budget
  • Budget amount: $10 per month
  • Alert threshold: 100% of budgeted amount
  • Email recipients: your email address
  • Click Create budget
Screenshot: Budgets page showing your $10 budget created
Step 5Create an IAM admin user

Go to IAM → Users → Create user:

  • User name: admin
  • Check Provide user access to the AWS Management Console
  • Select I want to create an IAM user
  • Set a strong password — uncheck Users must create a new password at next sign-in
  • Click Next

On the permissions page:

  • Select Attach policies directly
  • Search for and attach AdministratorAccess
  • Click Next → Create user
Also enable MFA on this user — go to the user → Security credentials → Assign MFA device. Because this is an admin account, protect it the same way as root.
Screenshot: IAM user created with AdministratorAccess policy attached
Step 6Sign out of root and sign in as IAM admin

Sign out of the root account. Sign in using your account ID and the admin username:

  • Go to: https://[your-account-id].signin.aws.amazon.com/console
  • Username: admin — Password: the password you just set

Confirm you are logged in as the IAM user — the top-right menu should show admin @ [account-id], not your email address.

From this point forward, always sign in as your IAM admin user. Never use root for day-to-day work.
Screenshot: Console showing you are signed in as the IAM admin user (top-right corner)
🔵 Do the following steps as IAM ADMIN
Step 7Create a CloudTrail trail

Go to CloudTrail → Trails → Create trail:

  • Trail name: photoviewer-trail
  • Storage location: Create new S3 bucket — accept the auto-generated name
  • Leave all other settings as default → Next → Next → Create trail
CloudTrail logs every API call in your account — including calls made through the console UI. This is your audit trail for the entire course. It applies to all regions automatically.
Screenshot: CloudTrail showing photoviewer-trail with status Logging
Step 8Create a CloudWatch billing alarm
Make sure your region is set to US East (N. Virginia) — us-east-1. Billing metrics only exist in this region. If you are in any other region, you will not see the billing metric.

Go to CloudWatch → Alarms → Create alarm → Select metric → Billing → Total Estimated Charge:

  • Select EstimatedCharges with Currency: USD → Select metric
  • Threshold: Greater than 10
  • Click Next

Notification:

  • Alarm state trigger: In alarm
  • Create new SNS topic: billing-alerts
  • Email endpoint: your email address → Create topic
  • Check your email and confirm the subscription

Alarm name: billing-alarm-10 → Create alarm

You now have two billing alerts: the Budget alarm (sent to root) and this CloudWatch alarm (sent to your IAM admin email). Both alert you at $10.
Screenshot: CloudWatch alarm showing billing-alarm-10 with $10 threshold
Step 9Verify with CloudShell

Click the >_ icon in the top navigation bar to open CloudShell. Run each command and confirm the expected output.

Confirm you are logged in as IAM admin (not root)

aws sts get-caller-identity

The Arn field should show your IAM user — not root:

{"Arn": "arn:aws:iam::123456789012:user/admin", ...}

Confirm CloudTrail is enabled

aws cloudtrail describe-trails \
  --region us-east-1 \
  --query 'trailList[].[Name,S3BucketName,IsLogging]' \
  --output table

You should see photoviewer-trail in the output.

Confirm billing alarm exists

aws cloudwatch describe-alarms \
  --region us-east-1 \
  --query 'MetricAlarms[].[AlarmName,StateValue,Threshold]' \
  --output table

You should see billing-alarm-10 with a threshold of 10.

Screenshot: CloudShell showing output of all three commands
Step 10Clone the course repo

Clone the course GitHub repository into CloudShell. You only need to do this once.

git clone https://github.com/chalasanisree-courses/aws-security-course.git

Run the resource audit script to confirm it works:

bash aws-security-course/photoviewer/audit-aws-resources.sh

At this point you should have no expensive resources running.

Screenshot: CloudShell showing successful git clone and audit script output

Reflection

Answer both questions in 2–3 sentences each.

Question 1

Why should you not use the root account for day-to-day work, even though it has full access to everything? What specific risk does this mitigate?

Question 2

CloudTrail logs every API call in your account — including calls made through the console UI. Why is this important for security, and what would an attacker's activity look like in CloudTrail logs?
