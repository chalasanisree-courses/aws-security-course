Week 1 Lab — AWS Account Setup

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  25 pts  ·  Due Apr 9

Objective: Set up your AWS account with the security baseline required for this course — MFA on root, a dedicated IAM admin user, CloudTrail logging across all regions, and a billing alarm. These controls stay in place for the entire course.

Important: Use your own personal AWS account for this course — not a shared or employer account. All labs involve creating and deleting resources, and you need full admin access. AWS Free Tier is sufficient for most of this course.

Part 1 — Secure the root account

Step 1Enable MFA on root

The root account has unrestricted access to everything in your AWS account. It must be protected with MFA and then not used for day-to-day work.

Sign in to the AWS Console with your root account (the email address you used to create the account). Go to the top-right menu → Security credentials:

  • Scroll to Multi-factor authentication (MFA) → click Assign MFA device
  • Device name: root-mfa
  • MFA device type: Authenticator app (Google Authenticator, Authy, or similar)
  • Scan the QR code with your authenticator app → enter two consecutive codes → Add MFA
Screenshot: Security credentials page showing MFA device assigned to root

Part 2 — Create an IAM admin user

Step 2Create the admin user

You should never use the root account for daily work. Create a dedicated IAM admin user for everything in this course.

Go to IAM → Users → Create user:

  • User name: admin
  • Check Provide user access to the AWS Management Console
  • Select I want to create an IAM user
  • Console password: set a strong password
  • Uncheck Users must create a new password at next sign-in
  • Click Next

On the permissions page:

  • Select Attach policies directly
  • Search for and select AdministratorAccess
  • Click Next → Create user
Copy the sign-in URL shown after creation — it will look like https://[account-id].signin.aws.amazon.com/console. Bookmark it.
Screenshot: IAM user created with AdministratorAccess policy attached
Step 3Sign in as the IAM admin user

Sign out of the root account. Sign in using the IAM user sign-in URL you just copied, using your admin username and password.

Confirm you are logged in as the IAM user — the top-right menu should show admin @ [account-id], not your email address.

From this point forward, always sign in as your IAM admin user — never as root. Root should only be used for account-level tasks that specifically require it.
Screenshot: AWS Console showing you are signed in as the IAM admin user (top-right corner)

Part 3 — Enable CloudTrail

Step 4Create an all-regions trail

CloudTrail logs every API call made in your account. This is your audit trail — essential for security investigation and compliance.

Go to CloudTrail → Trails → Create trail:

  • Trail name: photoviewer-trail
  • Storage location: Create new S3 bucket — accept the auto-generated name
  • Log file SSE-KMS encryption: leave unchecked (not required for this course)
  • CloudWatch Logs: leave unchecked
  • Click Next

On the Events page:

  • Event type: Management events (default)
  • API activity: Read and Write
  • Click Next → Create trail
By default, a trail created in the console applies to all regions automatically. This is what you want — API calls in any region are logged.
Screenshot: CloudTrail showing photoviewer-trail with status Logging

Part 4 — Create a billing alarm

Step 5Enable billing alerts

First, enable billing alerts so CloudWatch can access billing data.

Go to Billing and Cost Management → Billing preferences:

  • Check Receive CloudWatch billing alerts
  • Click Save preferences
Step 6Create the alarm
Billing metrics are only available in us-east-1. Make sure your region is set to US East (N. Virginia) before proceeding.

Go to CloudWatch → Alarms → Create alarm:

  • Click Select metric → Billing → Total Estimated Charge
  • Select EstimatedCharges with Currency: USD → click Select metric
  • Threshold type: Static
  • Condition: Greater than → threshold value: 10
  • Click Next

On the notification page:

  • Alarm state trigger: In alarm
  • Select Create new topic
  • Topic name: billing-alerts
  • Email endpoint: enter your email address
  • Click Create topic → check your email and confirm the subscription
  • Click Next

Alarm name: billing-alarm-$10 → Next → Create alarm

Screenshot: CloudWatch alarm showing billing-alarm-$10 with $10 threshold

Part 5 — Verify with CloudShell

Step 7Open CloudShell and verify your setup

Click the >_ icon in the top navigation bar of the AWS Console to open CloudShell. Run each command and confirm the expected output.

Confirm you are logged in as IAM admin (not root)

aws sts get-caller-identity

The Arn field should show your IAM admin user — not root:

{"Arn": "arn:aws:iam::123456789012:user/admin", ...}

Confirm CloudTrail is enabled

aws cloudtrail describe-trails \
  --region us-east-1 \
  --query 'trailList[].[Name,S3BucketName,IsLogging]' \
  --output table

You should see photoviewer-trail in the output.

Confirm billing alarm exists

aws cloudwatch describe-alarms \
  --region us-east-1 \
  --query 'MetricAlarms[].[AlarmName,StateValue,Threshold]' \
  --output table

You should see billing-alarm-$10 with a threshold of 10.

Screenshot: CloudShell showing output of all three commands
Step 8Clone the course repo

Clone the course GitHub repository into CloudShell. You only need to do this once — CloudShell persists your home directory between sessions.

git clone https://github.com/chalasanisree-courses/aws-security-course.git

Then run the resource audit script to confirm it works:

bash aws-security-course/photoviewer/audit-aws-resources.sh

At this point you should have no expensive resources running — the output should show nothing billable.

Screenshot: CloudShell showing successful git clone and audit script output
