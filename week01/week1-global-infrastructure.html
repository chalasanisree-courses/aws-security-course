AWS Global Infrastructure

Understanding regions, availability zones, and how your AWS account relates to the physical infrastructure underneath it.

Learning objectives

  • Explain the difference between AWS regions and availability zones
  • Describe what an AWS account is — and what it is not
  • Identify which AWS services are regional vs. global
  • Explain why CloudWatch billing alarms are only available in us-east-1

Two things that are easy to confuse

AWS infrastructure and your AWS account are completely separate concepts that are easy to conflate early on. Understanding the difference is foundational — it explains a number of AWS behaviors that otherwise seem arbitrary.

AWS infrastructure is physical. It is data centers, fiber cables, networking equipment, and power systems that Amazon owns and operates around the world. You have no control over it and never touch it directly. It is organized into regions and availability zones.

Your AWS account is a logical construct — a billing and identity boundary that exists in AWS's management systems. It is not a physical thing and has no location. It spans whichever regions you choose to deploy services in, and you receive one bill aggregating usage across all of them.

The key insight: When you launch an EC2 instance in us-west-2, that instance runs on physical hardware in Oregon. But your account — the entity that owns the instance and gets billed for it — has no physical location. It is a record in AWS's systems that happens to have a resource in Oregon.

Key concepts at a glance

ConceptWhat it isExample
Region A geographically distinct cluster of AWS data centers. Each region is fully independent — a failure in one does not affect others. us-east-1 (N. Virginia), eu-west-1 (Ireland)
Availability Zone One or more physically separate data centers within a region, connected by low-latency private links. A failure in one AZ does not affect another. us-east-1a, us-east-1b, us-east-1c
AWS Account A logical billing and identity boundary. Not a physical location. One account can have resources in many regions and receives one monthly bill. Your 12-digit account ID — top-right console menu
Regional service A service where each resource exists in a specific region and is isolated from the same service in other regions. EC2, S3, Lambda, VPC, RDS, DynamoDB
Global service A service that operates across all regions from a single control plane. No region selector is needed. IAM, Route 53, CloudFront, Billing, AWS Organizations

Why CloudWatch billing alarms only work in us-east-1

Your AWS bill is account-level — it aggregates usage across every region into a single monthly total. There is no separate bill per region. Because billing is account-level rather than region-level, AWS treats it as a global concept.

However, CloudWatch — the monitoring service where you create alarms — is a regional service. Each region has its own CloudWatch instance. AWS made a design decision to publish billing metrics to only one CloudWatch region: us-east-1. The billing metric namespace simply does not exist in any other region's CloudWatch.

What this means in practice

It does not matter which region you normally work in. Even if all your application resources are in us-west-2, you must switch your console to us-east-1 to create a CloudWatch billing alarm. There is no workaround — the metric does not exist in other regions.

The steps are:

  • First enable billing alerts in Billing Preferences — a one-time account-level setting
  • Switch the console region selector to US East (N. Virginia) — us-east-1
  • Create the CloudWatch alarm in us-east-1
  • Switch back to your working region when done
Common mistake: Searching for the billing metric in your working region and seeing nothing. This is not a permissions problem — the metric genuinely does not exist outside us-east-1. Always check the region selector first when a CloudWatch metric seems to be missing.
Why us-east-1? It is the oldest AWS region and historically where AWS anchors account-level features. This is an AWS architectural convention worth knowing rather than reasoning about. You will encounter a few other features with the same behavior later in this course.

Resilience tiers — AZ tolerance vs regional tolerance

AWS does not automatically make your application resilient. Resilience is a spectrum with three tiers, and every tier is entirely the customer's responsibility to architect and enable. AWS guarantees the infrastructure within each region — it does not guarantee your application survives a regional failure.

The Photo Viewer application in this course runs as Tier 1 — single AZ — because it is a learning environment. This is a deliberate simplification. In a production environment handling real user data, Tier 2 at minimum would be expected.

The Middle East outage lesson: Customers who had only deployed in that region were simply down until AWS restored service. Customers with Tier 3 architecture failed over automatically. AWS was not at fault — they restored the region. Customers with no multi-region architecture had no recourse except to wait.

SCS-C03 exam note: The exam tests the Shared Responsibility Model in the context of global infrastructure. AWS is responsible for the physical security of data centers and the resilience of availability zones. You are responsible for choosing regions, designing for multi-AZ resilience, and ensuring data residency compliance for your workloads. Exam scenario questions that mention "natural disaster" or "99.99% availability" always point toward multi-region Tier 3 architecture.
