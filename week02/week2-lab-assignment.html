Week 2 Lab — Static S3 Hosting

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  25 pts  ·  Due Apr 16

Objective: Deploy the Photo Viewer as a static website directly from an S3 bucket. This is the deliberately insecure starting point — every vulnerability introduced here has a specific fix in Week 3.

Files for this lab: Get all files from the course GitHub repository → photoviewer/week02

Part 1 — Create S3 bucket

Step 1Create the bucket

Go to S3 → Buckets → Create bucket:

  • Bucket name: photoviewer-[your-account-id] — use your 12-digit AWS account ID to make it globally unique. Find your account ID in the top-right corner of the console.
  • AWS Region: us-east-1
  • Leave all other settings as default
  • Click Create bucket
Screenshot: S3 bucket created and showing in the bucket list

Part 2 — Enable static website hosting

Step 2Enable website hosting

Go to your bucket → Properties tab → scroll to Static website hosting → click Edit:

  • Static website hosting: Enable
  • Hosting type: Host a static website
  • Index document: index.html
  • Click Save changes

Part 3 — Make the bucket public (deliberately insecure)

Note: You are intentionally making this bucket public to demonstrate the vulnerability. This is not how production buckets should be configured — we fix this in Week 3.
Step 3Disable Block Public Access

Go to your bucket → Permissions tab → Block public access → click Edit:

  • Uncheck all four boxes
  • Click Save changes → type confirm → Confirm
Step 4Create bucket policy

Still on the Permissions tab → Bucket policy → click Edit → paste the following, replacing [your-account-id]:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadGetObject",
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::photoviewer-[your-account-id]",
      "arn:aws:s3:::photoviewer-[your-account-id]/*"
    ]
  }]
}

Click Save changes.

s3:ListBucket is included intentionally — this allows bucket enumeration, which is one of the vulnerabilities demonstrated at the end of this lab.
Screenshot: Permissions tab showing Block Public Access is OFF and bucket policy saved

Part 4 — Upload files

Step 5Add your photos

Go to your bucket → click Create folder → name it photos → Create folder.

Open the photos folder → click Upload → add 5 photo files named photo1.jpg through photo5.jpg → Upload.

If you use different filenames, update photos.json to match — the s3_key field must match the actual filename in S3.
Step 6Upload website files

Go to your bucket root (not inside the photos folder) → click Upload → add all of these files from the GitHub repo:

  • index.html
  • style.css
  • app.js
  • photos.json
  • confidential.txt

Click Upload.

confidential.txt must be at the bucket root alongside index.html — this is the file that demonstrates the vulnerability.
Screenshot: S3 bucket root showing all 5 files uploaded

Part 5 — Verify and demonstrate vulnerabilities

Step 7Access the site

Go to your bucket → Properties tab → scroll to Static website hosting → copy the Bucket website endpoint. It will look like:

http://photoviewer-[your-account-id].s3-website-us-east-1.amazonaws.com

Open it in your browser — the Photo Viewer should load and display your photos.

Screenshot: Photo Viewer loading in browser showing HTTP in address bar (no padlock)
Step 8Demonstrate the vulnerabilities

Vulnerability 1 — Access confidential.txt directly

Navigate to:

http://photoviewer-[your-account-id].s3-website-us-east-1.amazonaws.com/confidential.txt

The file contents are readable without any authentication.

Vulnerability 2 — Enumerate the bucket

In CloudShell, run:

aws s3 ls s3://photoviewer-[your-account-id] --no-sign-request

This lists all files in the bucket without logging in — including files you never intended to expose.

Vulnerability 3 — Confirm HTTP only

Look at the URL in the browser — it starts with http://. Try changing it to https:// — it will fail. The S3 website endpoint has no TLS support.

Screenshot: Terminal showing bucket enumeration output from --no-sign-request

Part 6 — Clean up

Step 9Run the audit script

In CloudShell:

cd ~/aws-security-course && git pull
bash photoviewer/audit-aws-resources.sh

S3 buckets are included in the output. Note: S3 storage is essentially free at this scale — no cleanup needed for the bucket itself. We will reuse it in Week 3.

Reflection

Answer both questions in 2–3 sentences each.

Question 1

You ran aws s3 ls s3://photoviewer-[your-account-id] --no-sign-request without any credentials and saw all the files in the bucket. Explain why this works — what specific AWS configuration choices made this possible?

Question 2

The S3 website endpoint is HTTP only — there is no way to enable HTTPS at this endpoint. Why is this a security problem even for a site that only shows public photos? Think about what future versions of this application will contain.
