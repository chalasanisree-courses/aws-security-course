Week 2 Lab — Static S3 Hosting

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  50 pts

Objective: Deploy the Photo Viewer as a static website directly from an S3 bucket. This is the deliberately insecure starting point — every vulnerability introduced here has a specific fix in Week 3.

Files for this lab: Get all files from the course GitHub repository → photoviewer/week02

Part 1 — Create S3 bucket

Step 1Create the bucket

Make sure you are signed in as your IAM admin user and your region is set to us-east-1.

Go to S3 → Buckets → Create bucket:

  • Bucket name: photoviewer-[your-account-id] — use your 12-digit AWS account ID to make it globally unique. Find it in the top-right corner of the console.
  • AWS Region: us-east-1
  • Block Public Access: uncheck "Block all public access" — and acknowledge the warning that appears
  • Leave all other settings as default → Create bucket
You are intentionally making this bucket public to host a website. This is not how production buckets should be configured — we fix this in Week 3.
Screenshot: S3 bucket created and showing in the bucket list

Part 2 — Upload all files

Step 2Upload all website files and photos

Go to your bucket → click Upload → add all of the following files from the GitHub repo:

  • index.html
  • style.css
  • app.js
  • photos.json
  • confidential.txt
  • All 5 photos (photo1.jpg through photo5.jpg) — upload these into a folder called photos/
Upload the photos into a photos/ folder: click Add folder during upload and name it photos, then add your photo files inside it. The app.js file references them at the path photos/photo1.jpg etc.
confidential.txt must be at the bucket root alongside index.html — this is the file that demonstrates the public bucket vulnerability.
Screenshot: S3 bucket showing all files uploaded including the photos/ folder

Part 3 — Enable static website hosting

Step 3Enable website hosting

Go to your bucket → Properties tab → scroll to Static website hosting → click Edit:

  • Static website hosting: Enable
  • Hosting type: Host a static website
  • Index document: index.html
  • Click Save changes

Part 4 — Add bucket policy

Step 4Create bucket policy to allow public access

Go to your bucket → Permissions tab → Bucket policy → click Edit → paste the following, replacing [your-account-id]:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadGetObject",
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::photoviewer-[your-account-id]",
      "arn:aws:s3:::photoviewer-[your-account-id]/*"
    ]
  }]
}

Click Save changes.

s3:ListBucket is included intentionally — this allows bucket enumeration, which is one of the vulnerabilities you will demonstrate below.
Screenshot: Permissions tab showing bucket policy saved

Part 5 — Verify and demonstrate vulnerabilities

Step 5Access the website

Go to your bucket → Properties tab → scroll to Static website hosting → copy the Bucket website endpoint. It will look like:

http://photoviewer-[your-account-id].s3-website-us-east-1.amazonaws.com

Open it in your browser — the Photo Viewer should load and display your photos.

Screenshot: Photo Viewer loading in browser showing http:// in address bar (no padlock)
Step 6Demonstrate the vulnerabilities

Vulnerability 1 — Access confidential.txt directly

Navigate to:

http://photoviewer-[your-account-id].s3-website-us-east-1.amazonaws.com/confidential.txt

The file contents are readable without any authentication.

Vulnerability 2 — Enumerate the bucket without credentials

In CloudShell, run:

aws s3 ls s3://photoviewer-[your-account-id] --no-sign-request

This lists all files in the bucket without logging in — including files you never intended to expose.

Vulnerability 3 — HTTP only, no HTTPS

The URL starts with http://. Try changing it to https:// — it will fail. The S3 website endpoint has no TLS support.

Screenshot: confidential.txt readable in browser
Screenshot: CloudShell showing bucket enumeration output from --no-sign-request

Part 6 — CloudTrail investigation

Step 7Check what CloudTrail logged

Go to CloudTrail → Event history. Filter by resource name — paste your bucket name.

You should see management events: bucket creation, bucket policy, static website configuration. But you will not see the GetObject call that retrieved confidential.txt.

CloudTrail logs management events by default — API calls that create, modify, or delete resources. It does not log data events (GetObject, PutObject) unless you explicitly enable them. This means an attacker reading your confidential file leaves no trace in the default setup.
Step 8Enable data events, test, then disable
Data events cost money — enable them only for this test, then disable them immediately after.

Go to CloudTrail → Trails → photoviewer-trail → Edit:

  • Under Data events → click Add data event type → S3
  • Specify your bucket: browse and select photoviewer-[your-account-id]
  • Log both Read and Write events → Save changes

Now access confidential.txt again in your browser. Wait ~2 minutes, then go back to CloudTrail → Event history and filter by your bucket name. You should now see a GetObject event for confidential.txt.

After verifying, disable data events immediately: CloudTrail → Trails → photoviewer-trail → Edit → remove the S3 data event → Save changes.

Screenshot: CloudTrail event history showing GetObject event for confidential.txt

Part 7 — Run the audit script

Step 9Check your running resources

In CloudShell:

cd ~/aws-security-course && git pull
bash photoviewer/audit-aws-resources.sh

S3 is essentially free at this scale — no cleanup needed. You will reuse this bucket in Week 3.

Reflection

Answer both questions in 2–3 sentences each.

Question 1

You ran aws s3 ls s3://photoviewer-[your-account-id] --no-sign-request without any credentials and saw all the files in the bucket. Explain why this works — what specific AWS configuration choices made this possible?

Question 2

The S3 website endpoint is HTTP only — there is no way to enable HTTPS at this endpoint. Why is this a security problem even for a site that only shows public photos? Think about what future versions of this application will contain.
