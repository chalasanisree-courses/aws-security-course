S3 Object Storage and Security

Hosting the Photo Viewer on AWS for the first time — using S3 static website hosting — and understanding the security tradeoffs that come with it.

Learning objectives
  • Explain what object storage is and how S3 differs from block and file storage
  • Describe how S3 static website hosting works and what it cannot do
  • Explain the three layers of S3 access control and how they are evaluated
  • Describe how data is encrypted in transit and at rest in S3
  • Identify the security problems introduced by this approach and what fixes them

1. Starting point

In Week 1 we set up our AWS account, created an IAM admin user, enabled CloudTrail, and configured billing alarms. We have an AWS account — but no application yet.

The Photo Viewer is a web application that displays photos. The first version is intentionally simple: five photos served from a hardcoded list, HTML and JavaScript running entirely in the browser, no backend. We need somewhere to store and serve HTML, CSS, JavaScript, and image files. S3 static website hosting does exactly this — zero servers, zero configuration overhead.

Week 2 is deliberately insecure. We build the simplest possible version first so we can see exactly what's wrong with it. Every vulnerability introduced this week has a specific fix in a later week.

2. The solution — S3 static website hosting

S3 is AWS's object storage service — a managed file store where you put objects (files) into buckets (containers) and access them via URLs. Enabling static website hosting tells S3 to behave like a basic web server: serve index.html on root requests, respond to GET requests with the matching file.

No servers to launch. No OS to patch. No capacity to plan. All files — index.html, style.css, app.js, photos.json, and the photos — live in the bucket. The browser fetches photos.json to get the photo list, then requests each image. S3 serves everything. All logic runs in the browser — there is no backend.

3. Architecture

Week 2 is the simplest possible architecture — a single S3 bucket serving a public website directly to browsers over HTTP.

4. How it works — request walkthrough

Step 1
User opens the Photo Viewer URL

Browser sends an HTTP GET to the S3 website endpoint. S3 returns index.html, the browser begins rendering, then requests style.css and app.js.

Step 2
JavaScript fetches the photo list

app.js runs and calls fetch('/photos.json'). S3 returns the static JSON file — [{"photo_id":"photo-001","s3_key":"photos/photo1.jpg"}, ...]. This is a file, not a database query. The list is fixed at deploy time.

Step 3
Browser requests each photo

app.js loops through the list and requests each photo by its s3_key. S3 serves each image directly. The entire interaction is browser ↔ S3 — no backend server at any point.

📦Object storageWhat S3 is and how it differs from block and file storage
›

There are three ways to store data in the cloud:

TypeHow it worksAWS serviceBest for
BlockFixed-size blocks on a disk, assembled by the OS file systemEBSOS disks, databases
FileNetwork file system multiple servers can mount simultaneouslyEFSShared content across EC2 instances
ObjectDiscrete objects retrieved via direct API call — no OS file systemS3Photos, video, backups, static web files

The application talks directly to S3 via API. S3 returns the object to whoever requested it — no file system in between.

🪣S3 bucketphotoviewer-9876543210 · us-east-1 · globally unique name
›

A bucket is a container for objects. Key properties: bucket names are globally unique across all AWS accounts (that's why ours includes the account ID); each bucket lives in a specific region; S3 automatically replicates across multiple AZs; S3 exists outside your VPC.

Objects are accessed via URL: https://bucket-name.s3.us-east-1.amazonaws.com/path/to/object.jpg

🌐Static website hostingServes index.html on root · HTTP only · no server-side logic
›

Enabling static website hosting gives the bucket a web endpoint: http://bucket.s3-website-us-east-1.amazonaws.com.

What it does: Serves index.html on root requests, returns matching files for any path, returns a 404 error page when a file isn't found.

What it does NOT do: No server-side code (no PHP, Python, Node.js). No SSL/TLS — HTTP only. No sessions, cookies, or database connections.

Mental model: S3 is a smart file server. All logic lives in the browser JavaScript. S3 just delivers files when asked.

🔐Access control3 layers: Block Public Access · bucket policy · object ACL
›

Every S3 request passes through three layers in order. All must allow for access to succeed.

LayerWhat it controlsWeek 2 setting
Block Public AccessPrevents access from Principal: * (anonymous internet). On by default on new buckets.Disabled — required for static hosting
Bucket policyExplicit deny overrides all. Explicit allow grants. No mention → check ACL.s3:GetObject + s3:ListBucket to Principal: *
Object ACLPer-object control. Legacy — use bucket policies instead.Default

The consequence: granting Principal: * exposes every file — not just web files. This is one of the Week 2 vulnerabilities.

🔒EncryptionAt rest: SSE-S3 automatic · In transit: HTTP only (no TLS)
›

Data at rest: S3 encrypts all objects automatically using SSE-S3 — AWS manages the key. Happens by default, no configuration needed.

TypeWho manages keyWhen to use
SSE-S3AWS — automaticDefault — most use cases
SSE-KMSYou — via AWS KMSCompliance, key usage audit logs
Client-sideYou — outside AWSData must never reach AWS unencrypted

Data in transit: The S3 website endpoint is HTTP only — no TLS. Content travels unencrypted. Fixed in Week 3 by CloudFront.

📋CloudTrail loggingManagement events logged · data events optional · anonymous access leaves a trace
›

Even with a public bucket, access isn't invisible. CloudTrail can log S3 data events including anonymous GetObject and ListBucket calls — the userIdentity field shows Anonymous, which is a signal worth alerting on.

Important: S3 data events are not logged by default — only management events (creating buckets, changing policies) are. Without explicit data event logging enabled, file-level access leaves no trace.

Try it: aws s3 ls s3://photoviewer-9876543210 --no-sign-request lists every file in the bucket. Then aws s3 cp s3://photoviewer-9876543210/confidential.txt . downloads it — no credentials needed. This is what "public bucket" means in practice.

Issue 1 — HTTP only

The S3 website endpoint is HTTP only. Content travels in plaintext — anyone on the same network can read or modify it in transit. Fixed in Week 3 by CloudFront, which enforces HTTPS and redirects HTTP automatically.

Issue 2 — Public bucket

Static website hosting requires Block Public Access to be disabled and Principal: * on the bucket policy. Every file is publicly accessible — including confidential.txt. Anyone can enumerate all files with aws s3 ls --no-sign-request. Fixed in Week 3 by Origin Access Control — bucket goes fully private, only CloudFront can read it.

Issue 3 — No caching

The bucket is in us-east-1. Every request from every user worldwide travels to Northern Virginia and back. For a real application serving global users, this latency and cost per request are unacceptable. Fixed in Week 3 by CloudFront — content cached at 400+ edge locations globally.
