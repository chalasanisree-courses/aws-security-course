S3 Object Storage and Security

How AWS S3 works, how access is controlled, how data is protected, and how S3 is used as a web server — and why that creates security challenges.

Learning objectives

  • Explain what object storage is and how it differs from block storage
  • Describe what an S3 bucket is and how objects are stored and accessed
  • Explain the three layers of S3 access control and how they are evaluated
  • Describe data in motion vs data at rest and how each is encrypted in S3
  • Explain how S3 static website hosting works and what it cannot do
  • Identify the security problems introduced by hosting a public website directly from S3

1. What is object storage?

There are three ways applications store data in the cloud. Understanding the differences explains why S3 exists and when to use it.

Storage type How it works AWS service Best for
Block storage Data stored as fixed-size blocks on a disk. The OS has a file system that locates blocks and assembles them into files for the application. EBS — Elastic Block Store OS disks, databases, anything that needs low-latency random read/write
File storage A network file system that multiple servers can mount simultaneously. The file system lives on the network rather than on each server. EFS — Elastic File System Shared content that multiple EC2 instances need to access simultaneously
Object storage Data stored as discrete objects. The application makes a direct API call to retrieve an object — no OS file system involved, no concept of blocks. S3 — Simple Storage Service Photos, videos, documents, backups, static website files — anything accessed via HTTP
The key difference with object storage: The application talks directly to S3 via an API call. There is no operating system file system in between. S3 returns the object — a photo, a document, a JavaScript file — directly to whoever requested it.

2. S3 — buckets and objects

Buckets

A bucket is a container for objects. Every object in S3 lives inside a bucket. Key facts about buckets:

  • Bucket names are globally unique across all AWS accounts worldwide — if someone else has taken a name, you cannot use it
  • Each bucket lives in a specific AWS region — you choose the region when you create it
  • AWS automatically stores your data across multiple availability zones within that region — you don't configure this, it's built in
  • You don't choose how many servers store your data — S3 handles all of that invisibly

Objects

An object is any file stored in S3 — a photo, a PDF, a JavaScript file, a video, a database backup. There is no practical limit on file type or size (up to 5TB per object). Objects are accessed via a URL:

https://bucket-name.s3.region.amazonaws.com/path/to/object.jpg

S3 is not inside your VPC

S3 is an AWS managed service that exists outside your Virtual Private Cloud. You access it via API calls over the internet or via a VPC endpoint. AWS manages all the underlying infrastructure — servers, disks, networking, replication. You manage the bucket, what's in it, and who can access it.

Regional scope, automatic resilience: Your S3 bucket is a regional resource — it lives in the region you chose. But unlike EC2 which is AZ-specific, S3 is automatically resilient within the region. AWS guarantees your data is accessible even if one or more AZs within that region go down. You don't configure this.

Block Public Access only affects anonymous users

Block Public Access is specifically designed to prevent access from Principal: * — anonymous internet users with no AWS credentials. It has no effect on authenticated IAM users. If an IAM user with the right permissions tries to access an S3 object, Block Public Access does not interfere.

The guard and clipboard analogy

Think of the bucket policy as a guard at the door with a clipboard. An explicit deny means the requester is on the blacklist — turned away regardless of anything else. An explicit allow means they're on the whitelist — let through. No mention means the guard steps aside and the object ACL decides.

Object ACLs are legacy

AWS recommends bucket policies over object ACLs. A bucket policy applies once to the entire bucket. ACLs require per-object configuration — unmanageable at scale.

Default state: New S3 buckets have Block Public Access enabled by default. Even a bucket policy that allows public access will be blocked until you explicitly disable it. This is a deliberate safety default.

Encryption — data in motion and data at rest

Data has two states that require different encryption approaches. S3 handles both automatically — but understanding each matters when you need to go beyond the defaults.

Data in motion

When a browser or application requests an object from S3, the data travels over the internet. This is encrypted using TLS — Transport Layer Security — the foundation of HTTPS. Anyone intercepting the network traffic sees encrypted bytes, not the actual content.

Data at rest

Once an object is stored on S3 disk, it is data at rest. S3 automatically encrypts all data at rest using server-side encryption (SSE) — you don't configure this, it happens for every object by default.

Encryption type Who manages the key When to use
SSE-S3 AWS manages the key automatically Default — most use cases. You don't control the key.
SSE-KMS You create and manage the key in AWS KMS When compliance requires customer-managed keys or audit logs of every key usage
Client-side encryption You manage the key entirely outside AWS When data must never reach AWS unencrypted — highest sensitivity
Why does AWS encrypt by default? AWS stores data for millions of customers on shared physical infrastructure. If the physical hardware were ever compromised, encrypted data is protected even if an attacker gains access to the disks. Encryption at rest is a defense-in-depth measure — it protects data even when access controls fail.

4. S3 as a static web server

S3 has a feature called static website hosting that allows it to serve web content directly to browsers. Understanding exactly what S3 does — and doesn't do — in this mode is important.

What S3 does as a web server

  • Serves index.html automatically when a user requests the root URL
  • Serves a custom error page when a file is not found
  • Responds to HTTP GET requests by returning the requested file

What S3 does NOT do as a web server

  • No server-side code execution — no PHP, Python, Node.js, or any server-side logic
  • No SSL/TLS at the website endpoint — the S3 website URL is HTTP only
  • No sessions, no server-side cookies, no database connections
  • No request processing or business logic
The right mental model: S3 is a smart file server. It maps HTTP GET requests to objects in the bucket and returns them. All the intelligence in a static web application lives in the JavaScript running in the browser — not on S3. S3 just delivers the files.

How the Photo Viewer uses S3

The Photo Viewer is a static application. S3 stores and serves all the files — index.html, style.css, app.js, and the photos. When you click Next Photo, the JavaScript running in your browser cycles through the photo list and makes new requests to S3 to fetch each image. S3 has no knowledge that a button was clicked — it just responds to file requests.

CloudTrail still records everything

Even though this bucket is publicly accessible and anyone can read files without credentials, those accesses are not invisible. CloudTrail can be configured to log S3 data events — including anonymous GetObject and ListBucket calls from unauthenticated requesters.

This means an attacker who accesses confidential.txt directly, or enumerates the bucket with --no-sign-request, leaves a trace in your logs. The userIdentity field in the CloudTrail event will show Anonymous — which itself is a signal worth alerting on.

Important: S3 data events are not logged by default. CloudTrail only logs management events (creating buckets, changing policies) unless you explicitly enable data event logging on your trail. This is an extra cost — but without it, file-level access leaves no trace.
