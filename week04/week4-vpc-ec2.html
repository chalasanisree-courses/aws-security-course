VPC, EC2, and Application Tier Security

Adding a real backend to the Photo Viewer — a 3-tier architecture with a private EC2 app server, DynamoDB, ALB, and VPC — and tracing exactly what happens when the page loads.

Learning objectives
  • Explain why a static photo list is limiting and how a dynamic backend solves it
  • Map the 3-tier architecture pattern to AWS services in the Photo Viewer
  • Trace the full request path from browser to DynamoDB and back
  • Explain the purpose of each architectural component and why it is placed where it is
  • Identify the deliberate security vulnerabilities in the Week 4 architecture and their fixes

1. The problem with a static photo list

In Weeks 2 and 3 the Photo Viewer served a static photos.json file listing five photos. This works, but the list is fixed at deploy time. Adding a new photo to S3 means manually editing photos.json, re-uploading it, and invalidating the CloudFront cache. For an application where users can upload content this is unacceptable.

What we want: add a photo to S3, have it appear to all visitors immediately. That requires a backend that queries a database at runtime — not a file that was written at deploy time.

The one-line frontend change: Everything in app.js stays the same except one line — fetch('/photos.json') becomes fetch('/photos'). The browser now calls an API endpoint on our app server instead of fetching a static file. The backend does all the new work.

2. The solution — 3-tier application architecture

We introduce a real backend: an EC2 application server in a private VPC subnet, backed by DynamoDB. The photo list is now constructed dynamically at request time — add a row to DynamoDB and it appears to all users immediately.

Security principle: Each tier only communicates with the tier immediately adjacent to it. The browser never talks to DynamoDB directly. The app server never talks to the browser directly. This containment limits the blast radius if any one component is compromised.

🏗VPCphotoviewer-vpc · 10.0.0.0/16
›

A Virtual Private Cloud is a logically isolated network inside AWS — a fenced-off section of the datacenter that belongs only to your account. Your VPC spans two availability zones for resilience. Everything inside it is unreachable from the internet unless you explicitly open a path. The CIDR 10.0.0.0/16 gives you 65,536 private IP addresses to allocate across subnets.

🔲Subnets2 public · 2 private · across 2 AZs
›

Subnets are subdivisions of the VPC. The difference between public and private is entirely about routing — a public subnet has a route to the Internet Gateway, a private subnet does not.

SubnetAZCIDRWhat lives here
public1us-east-1a10.0.0.0/20ALB node, NAT GW (temp)
public2us-east-1b10.0.16.0/20ALB node (required for multi-AZ)
private1us-east-1a10.0.128.0/20App Server (EC2), SSM endpoint ENIs
private2us-east-1b10.0.144.0/20Empty — for symmetry and future use

Why EC2 is in the private subnet: Even if a security group was misconfigured, there is no network route from the internet to the private subnet. Subnet placement is the outer wall; security groups are the inner wall — defense in depth.

🔗Internet GatewayOutbound internet for public subnets · NAT GW uses it during bootstrap
›

The IGW is attached to the VPC and provides the internet connection for public subnets. The public route table has one key rule: 0.0.0.0/0 → IGW. This is what makes a subnet "public." Private subnets have no such route — they cannot initiate or receive internet connections directly. CloudFront does not go through the IGW — it connects directly to the ALB's public DNS over AWS's own backbone network.

⚖️Application Load BalancerBridges CloudFront to EC2 · health checks · spans both AZs
›

EC2 is in a private subnet with no public IP. CloudFront can only reach internet-routable endpoints. The ALB solves this — it sits in the public subnet with a public DNS name, receives requests from CloudFront, and forwards them to EC2's private IP inside the VPC.

The ALB also performs health checks — sending GET /health to port 8000 every 30 seconds. If the app server fails two consecutive checks, the ALB stops routing to it. This is why app.py has a /health endpoint returning 200.

Vulnerability: The ALB's DNS name is public — anyone who discovers it can call /photos directly, bypassing CloudFront and any WAF rules. Fix in Week 9: add a secret custom header that CloudFront sends, and configure the ALB to reject requests missing it.

🖥App Server (EC2)Private subnet · no public IP · app.py · systemd
›

EC2 runs app.py on port 8000, managed by systemd so it starts on boot and restarts on crash. It handles two endpoints: GET /health (200 OK for ALB) and GET /photos (scans DynamoDB, returns JSON). The instance has no public IP, no key pair, and port 22 is never open — all access is through SSM Session Manager.

On first boot, the user data script runs automatically: installs Python and packages via NAT Gateway, downloads app.py from S3 via the S3 Gateway endpoint, creates the systemd service, and starts it. After this bootstrap completes the NAT Gateway is deleted — EC2 has no internet access at runtime.

🔀NAT GatewayTemporary · bootstrap only · ~$32/month · delete immediately after
›

EC2 is on a private subnet with no internet access. But the user data script needs to run dnf update and pip3 install flask boto3 — both require reaching the public internet (Amazon Linux repos and PyPI). The NAT Gateway sits in the public subnet and provides outbound-only internet access for private subnet resources.

NAT is outbound only — the internet cannot initiate connections through it. After bootstrap, EC2 only needs DynamoDB (via Gateway endpoint) and SSM (via interface endpoints) — no internet required. So we delete the NAT Gateway immediately, saving ~$32/month and removing internet access from EC2 entirely.

🔐SSM Session ManagerRemote access without SSH · no port 22 · IAM-controlled · CloudTrail logged
›

The SSM Agent runs on EC2 and maintains a persistent outbound connection to the SSM service through three VPC interface endpoints. When you start a session from the console, your browser connects to SSM which relays it to the agent on EC2. No inbound ports needed — port 22 is never open.

Three interface endpoints are required — each creates an ENI with a private IP in the private subnet:

EC2 needs to reach AWS services — DynamoDB, S3, and SSM. Without endpoints, that traffic would go over the internet (requiring the NAT Gateway to stay running). Endpoints create private paths that stay inside AWS's network.

We have 5 endpoints total: 2 free Gateway endpoints (DynamoDB, S3) and 3 Interface endpoints for SSM. The 3 SSM interface endpoints cost ~$21/month combined — delete them when you're done with the lab.

🗄DynamoDBPhoto catalog · replaces static photos.json · enables per-user and public/private
›

DynamoDB is a fully managed NoSQL database — no servers to provision, no schema migrations, scales automatically. The photoviewer-photos table stores photo metadata with these attributes:

AttributeTypePurpose
photo_idString (partition key)Unique identifier e.g. photo-001
s3_keyStringPath to the photo in S3 — app server returns this so the browser knows what to request from CloudFront
is_publicBooleanApp server filters on is_public = true — private photos never appear in the list
ownerStringCurrently anonymous — Week 5 adds real user accounts and per-user filtering
uploaded_atString (ISO 8601)Timestamp for ordering

Why not static JSON? Adding a photo to photos.json requires editing, re-uploading, and a CloudFront cache invalidation. With DynamoDB, inserting a row is immediately visible to all users. Per-user and public/private filtering — impossible with a static file — are a simple query filter.

🔒Security groupsStateful firewalls · chained by SG ID · 3 groups forming a chain
›

Security groups are stateful firewalls attached to network interfaces. Each group in our chain references the previous one by security group ID — not by IP address. This means only resources with the correct security group attached can communicate, regardless of their IP.

Security groupInbound ruleSource
photoviewer-alb-sgHTTP :800.0.0.0/0 — CloudFront and internet
photoviewer-ec2-sgTCP :8000photoviewer-alb-sg only — no SSH, no port 22
photoviewer-endpoint-sgHTTPS :443photoviewer-ec2-sg only — SSM endpoints
Internet→:80→ ALB→:8000→ EC2→:443→ Endpoints→SSM
🚧Network ACLsStateless subnet-level firewall · default allows all · security groups do the real work here
›

Network ACLs (NACLs) are stateless packet filters that apply at the subnet boundary — before traffic even reaches a resource's security group. Unlike security groups, NACLs are stateless: you must explicitly allow both inbound and the return traffic outbound.

In Week 4 we use the default NACL which allows all inbound and outbound traffic. This is intentional — our security groups handle access control with more precision (by SG ID, not by IP). Adding custom NACL rules on top would add complexity without adding meaningful security for our architecture.

When NACLs do matter: Blocking a specific IP range across an entire subnet (e.g. blocking a country's IP range before it hits the ALB). Security groups cannot block by IP range at scale — NACLs can. This is a Week 9 topic when we add WAF and geo-restriction.

🪪IAM instance rolephotoviewer-ec2-role · 3 policies · no access keys on disk
›

Instead of storing access keys on the instance (which is insecure — keys can be stolen from disk or leaked in logs), we attach an IAM role. EC2 retrieves temporary credentials automatically from the instance metadata service. The credentials rotate automatically and expire — even if leaked they have a short window.

PolicyWhat it allows
AmazonSSMManagedInstanceCoreSSM Session Manager — remote access without SSH
photoviewer-dynamodb-readdynamodb:Scan on photoviewer-photos table only — no other tables, no writes
photoviewer-s3-app-reads3:GetObject on photoviewer-[account-id]/app/* only — just enough to download app.py at boot

This is least privilege — each permission is scoped to the exact resource and action needed. If EC2 is compromised, the attacker can only scan one DynamoDB table and read one S3 prefix. Nothing else.

🗺Route tablesPublic routes to IGW · private routes to Gateway endpoints · no 0.0.0.0/0 in private
›

Route tables control where traffic goes. Each subnet is associated with one route table. The critical difference between our public and private subnets is a single route table entry.

Route tableDestinationTargetMeaning
public10.0.0.0/16localVPC-internal traffic stays local
public0.0.0.0/0IGWEverything else goes to the internet
private10.0.0.0/16localVPC-internal traffic stays local
privatepl-XXXXX (DynamoDB)Gateway endpointDynamoDB traffic goes through private endpoint
privatepl-XXXXX (S3)Gateway endpointS3 traffic goes through private endpoint

The private route table has no 0.0.0.0/0 entry at runtime (the NAT Gateway route is deleted after bootstrap). This means there is literally no network path from the private subnet to the internet — not a firewall rule, an absent route. A misconfigured security group cannot override this.

4. How it works — request walkthrough

From the moment a user opens the Photo Viewer to the moment photos appear on screen — every component plays a role.

Step 1
Browser requests the page

User opens the Photo Viewer URL. The browser hits CloudFront, which forwards to S3 via OAC. CloudFront returns index.html, style.css, and app.js — served from cache if available.

Step 2
JavaScript requests the photo list

app.js makes a fetch('/photos') call. CloudFront has a cache behavior for this path — it routes the request to the ALB origin, not S3.

Step 3
CloudFront forwards to the ALB

CloudFront connects directly to the ALB's public DNS over AWS's backbone — not through the IGW. The ALB sits in the public subnet and accepts HTTP on port 80.

Step 4
ALB forwards to the app server

The ALB routes to EC2 on port 8000 via private IP. The EC2 security group only allows inbound :8000 from photoviewer-alb-sg — no other source can reach it.

Step 5
App server queries DynamoDB

app.py runs a DynamoDB Scan filtering for is_public = true. Traffic goes through the Gateway endpoint — stays inside AWS's network, no internet needed.

Step 6
App server returns JSON

The app server constructs [{"photo_id":"photo-001","s3_key":"photos/photo1.jpg"}, ...] and returns it up the chain: EC2 → ALB → CloudFront → browser.

Step 7
Browser fetches and displays photos

app.js loops through the JSON and requests each photo from CloudFront → S3 via OAC. Same path as Week 3. Photos appear on screen.

Verify in DevTools: Click the /photos request → Response Headers. You should see Server: Werkzeug/x.x Python/x.x (proves it came from EC2) and Via: CloudFront.

6. Problems with this solution

Click each vulnerability to see the attack scenario and fix

ALB is internet-facing — CloudFront bypassable
click to expand

The vulnerability: The ALB has a public DNS name (photoviewer-alb-XXXXXX.us-east-1.elb.amazonaws.com). Anyone who discovers this name can call the /photos API directly, bypassing CloudFront — and bypassing any WAF rules, geo-restrictions, or rate limiting at the CloudFront layer.

The fix: Add a secret custom header that CloudFront sends with every request to the ALB origin (e.g. X-Origin-Secret: abc123). Configure an ALB listener rule to return 403 for any request missing that header. This is the standard production pattern — AWS has no OAC equivalent for ALB origins.

Supply chain risk — pip installs from the public internet
click to expand

The vulnerability: The user data script runs pip3 install flask boto3 pulling whatever is currently published on PyPI. If a malicious version of boto3 was published, every new EC2 instance would install it. The malware runs as root, inherits the instance role, and can exfiltrate data.

Real example: In 2018 the npm package event-stream was taken over by a malicious actor who inserted a backdoor targeting cryptocurrency wallets — downloaded millions of times before discovery. Package managers are a high-value attack surface.

The fix: We already implemented least privilege — the role only allows dynamodb:Scan on one table and s3:GetObject on one prefix, so blast radius is contained. Additionally: pin package versions (pip3 install boto3==1.34.0), use a private package mirror (AWS CodeArtifact) so you control what's installable.

No rate limiting on /photos — Layer 7 DDoS risk
click to expand

The vulnerability: The /photos endpoint has no throttling. A flood of valid HTTP requests would exhaust the app server (single-threaded) and consume DynamoDB read capacity. Static content is safe because CloudFront serves it from cache — but API responses bypass the cache and hit EC2 every time.

L3/L4 vs L7 DDoS: Shield Standard protects against volumetric network-layer attacks automatically. But application-layer floods of valid HTTP requests look identical to real traffic at the network layer — Shield Standard cannot distinguish them. WAF rate limiting is the fix.

The fix: AWS WAF rate-based rule capping requests per IP per minute on the /photos path. We add WAF in Week 9.
