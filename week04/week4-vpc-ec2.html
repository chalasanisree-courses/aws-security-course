VPC, EC2, and Application Tier Security

Building a 3-tier application in AWS — presentation, application, and data tiers — with a private subnet EC2 backend, DynamoDB, ALB, VPC endpoints, and SSM access control.

Learning objectives
  • Explain the 3-tier application architecture and map it to AWS services
  • Design a VPC with public and private subnets and explain the security rationale for each
  • Explain how security groups chain together to implement least privilege at the network layer
  • Describe how SSM Session Manager provides remote access without SSH or open ports
  • Explain the difference between Gateway and Interface VPC endpoints
  • Explain why a NAT Gateway is needed for bootstrap but not for runtime
  • Explain why DynamoDB replaces static JSON and what the schema enables for future weeks
  • Identify the deliberate security vulnerabilities in the Week 4 architecture and their fixes

1. The problem with a static photo list

In Weeks 2 and 3 the Photo Viewer served a static photos.json file listing five photos. This works but has a fundamental limitation — the list is fixed at deploy time. Adding a new photo to S3 doesn't automatically show it to website visitors. You have to manually update photos.json and redeploy.

For a real application where users upload content this is unacceptable. We need the photo list to come from a database that can be updated at runtime without redeploying anything. That's what Week 4 solves.

The one-line change: The entire Week 4 frontend change is one line in app.js — replacing fetch('/photos.json') with fetch('/photos'). Everything else — the HTML, CSS, photo rendering logic — stays identical. The backend does the new work.

2. 3-tier application architecture

The Photo Viewer follows the classic 3-tier architecture pattern. Click any component to learn what it does.

Click any box to explore

The key security principle: Each tier only communicates with the tier immediately adjacent to it. The browser never talks to DynamoDB directly. EC2 never talks to the browser directly. Everything flows through defined interfaces — this containment limits the blast radius of any single component being compromised.

Why the app server returns JSON, not HTML

The application server (app.py, built using Flask) only returns JSON — [{"photo_id": "photo-001", "s3_key": "photos/photo1.jpg"}, ...]. The browser's JavaScript (app.js) receives this and renders the UI. The app server is purely application tier — no presentation responsibility.

3. VPC design

A VPC — Virtual Private Cloud — is a logically isolated network inside AWS. Think of it as a fenced-off section of the AWS datacenter that belongs only to your account. Click any component in the diagram to explore it.

Click any component to explore

ALB requires two subnets in different AZs. This is an AWS requirement — the ALB deploys a node in each selected AZ for availability. Even though we only have one EC2 instance in us-east-1a, the ALB needs both public subnets selected. The second subnet is there for the ALB node, not for EC2.

The security group chain

Each security group references the previous one by security group ID — not by IP range. This means only resources with the correct security group attached can communicate. Even if an attacker launched a new EC2 instance in the same VPC with any IP address, it cannot reach the app server unless it has photoviewer-alb-sg attached — which requires IAM permissions to modify security groups.

Internet → :80 → ALB (alb-sg) → :8000 → EC2 (ec2-sg) → :443 → Endpoints (endpoint-sg) → SSM service

5. SSM Session Manager — access without SSH

EC2 has no key pair and port 22 is never open. All access to the instance goes through AWS Systems Manager Session Manager.

How SSM works

The EC2 instance runs an SSM Agent (pre-installed on Amazon Linux). The agent maintains a persistent outbound connection to the SSM service through three VPC interface endpoints. When you start a session from the console, your browser connects to the SSM service which relays the session to the agent on EC2. No inbound ports needed.

Your browser → SSM service (AWS managed) ↕ (via ssmmessages endpoint) SSM Agent on EC2

Why three endpoints?

Endpoint Purpose
ssm Agent registration and heartbeat — EC2 announces itself to SSM, receives configuration
ssmmessages Interactive session traffic — your keystrokes and terminal output travel through this
ec2messages Run Command traffic — for sending shell commands without opening a full session

All three are required. If any one is missing the agent can't register or sessions won't connect.

SSM vs SSH — why SSM wins for security

SSH SSM Session Manager
Port 22 Must be open — constantly attacked by scanners Never needed
Key files Can be lost, stolen, shared No key files
Access control Whoever has the key file IAM policy — integrates with your identity system
Audit trail None by default Every session logged in CloudTrail automatically
Private subnet Requires bastion host Works natively via VPC endpoints

6. VPC endpoints

EC2 in a private subnet needs to reach AWS services — DynamoDB, S3, and SSM. Without VPC endpoints, that traffic would need to leave the VPC, go over the internet (requiring a NAT Gateway), and come back. VPC endpoints create private network paths from inside the VPC directly to AWS services.

Two types of endpoints

Gateway endpoint Interface endpoint
How it works A route table entry — like a road sign pointing traffic to AWS service An ENI (network interface) with a private IP inside your subnet — a physical connection point
Security group None needed — no ENI Required — controls who can reach the endpoint
Cost Free ~$0.01/hour per endpoint
Supported services S3 and DynamoDB only Any AWS service
We use it for DynamoDB, S3 SSM, ssmmessages, ec2messages

Gateway endpoints are route table entries. When you associate a Gateway endpoint with a route table, AWS adds an entry like pl-XXXXX (S3 prefix list) → vpce-XXXXX. Traffic destined for S3 or DynamoDB automatically takes this route — no DNS change, no application change needed.

7. NAT Gateway — temporary bootstrap only

EC2 in the private subnet has no internet access by default. But during first boot the user data script needs to install Python packages:

  • dnf update — downloads OS packages from Amazon Linux repositories — requires internet via NAT Gateway
  • pip3 install flask boto3 — downloads from PyPI — requires internet via NAT Gateway
  • aws s3 cp — downloads app.py from S3 — uses the S3 Gateway endpoint, no NAT Gateway needed

PyPI is not on AWS infrastructure — there is no VPC endpoint for it. The only way to reach PyPI from a private subnet is through a NAT Gateway.

NAT Gateway flow

EC2 (private subnet) → NAT Gateway (public subnet) → Internet Gateway → PyPI / Amazon Linux repos

NAT Gateway is outbound only — internet resources cannot initiate connections to your private subnet. It costs ~$32/month which is why we delete it immediately after EC2 bootstrap is complete. At runtime EC2 only needs DynamoDB (Gateway endpoint) and SSM (Interface endpoints) — no internet access required.

Bootstrap sequence

  1. EC2 starts — user data script runs automatically on first boot
  2. dnf update → downloads OS packages via NAT Gateway
  3. pip3 install flask boto3 → downloads from PyPI via NAT Gateway
  4. aws s3 cp → downloads app.py from S3 via S3 Gateway endpoint (no NAT needed)
  5. systemd service created → Flask starts → listens on port 8000
  6. NAT Gateway deleted — EC2 has zero internet access at runtime

systemd manages the app server. Instead of starting it with python3 app.py & (which dies on reboot), we create a systemd service. systemd starts the app server automatically whenever EC2 starts, and restarts it automatically if it crashes. This is the standard Linux pattern for managing long-running processes.

8. IAM instance role — least privilege

The EC2 instance needs permissions to call AWS services. Instead of storing access keys on the instance (which is insecure), we attach an IAM role. EC2 retrieves temporary credentials automatically from the instance metadata service.

The photoviewer-ec2-role has exactly three policies:

Policy Type What it allows
AmazonSSMManagedInstanceCore AWS managed SSM Session Manager access — required for remote access without SSH
photoviewer-dynamodb-read Customer inline dynamodb:Scan on photoviewer-photos table only — no other tables, no write access
photoviewer-s3-app-read Customer inline s3:GetObject on photoviewer-[account-id]/app/* only — just enough to download app.py at boot

This is least privilege — each permission is scoped to the exact resource and action needed. Nothing more.

9. Application Load Balancer

The ALB sits between CloudFront and EC2. It serves two purposes — routing and health checking.

Why an ALB instead of direct CloudFront → EC2?

EC2 is in a private subnet with no public IP. CloudFront is a global service outside your VPC — it can only reach internet-routable endpoints. The ALB is in the public subnet with a public DNS name, making it reachable from CloudFront. The ALB then routes the request to EC2's private IP inside the VPC.

Health checks

The ALB sends an HTTP GET to /health on port 8000 every 30 seconds. The app server responds with {"status": "ok"} and a 200 status code. If the health check fails twice consecutively the ALB marks the target unhealthy and stops routing traffic to it. This is why we have a /health endpoint in app.py.

Verifying the request path

In DevTools → Network → /photos request → Response Headers:

  • Server: Werkzeug/x.x.x Python/x.x.x — proves the response came from the app server on EC2 (Werkzeug is the development server built into Flask)
  • Via: 1.1 [id].cloudfront.net (CloudFront) — proves CloudFront forwarded the request
  • X-Cache: Miss from cloudfront — confirms caching is disabled for API responses

10. DynamoDB — the photo catalog

DynamoDB is a fully managed NoSQL database — no servers to provision, no schema migrations, scales automatically. We use it to store photo metadata so the photo list can be updated at runtime without redeploying anything.

Table schema

Attribute Type Purpose
photo_id String — partition key Unique identifier e.g. photo-001
s3_key String Path to the photo in S3 e.g. photos/photo1.jpg — the app server returns this so the browser can request the image from CloudFront
is_public Boolean Whether the photo is visible to anonymous users. The app server filters on is_public = true
owner String Username of the photo owner. Currently anonymous — Week 5 adds real user accounts
uploaded_at String (ISO 8601) Timestamp — for ordering and audit purposes

Why database instead of static JSON?

static photos.json (Week 2/3) DynamoDB (Week 4+)
Add a photo Edit JSON file, redeploy to S3, invalidate CloudFront cache Insert a row — visible immediately
Per-user photos Impossible — one static list for everyone Filter by owner field
Public/private Impossible — all photos visible to all Filter by is_public field
Scale Impractical beyond dozens of photos Millions of items, automatic scaling

12. Security vulnerabilities in Week 4

Click each vulnerability to see the attack scenario and fix

ALB is internet-facing — CloudFront bypassable
click to expand

The vulnerability: The ALB has a public DNS name (photoviewer-alb-XXXXXX.us-east-1.elb.amazonaws.com). Anyone who discovers this name can call the /photos API directly, bypassing CloudFront entirely — and bypassing any WAF rules, geo-restrictions, or rate limiting configured at the CloudFront layer.

The fix: Add a secret custom header that CloudFront sends with every request to the ALB origin (e.g. X-Origin-Secret: abc123). Configure an ALB listener rule to return 403 for any request missing that header. Without the secret, direct ALB access is blocked. This is the standard production pattern since AWS does not have an OAC equivalent for ALB origins.

Overprivileged instance role — supply chain attack vector
click to expand

The vulnerability: If an attacker achieves code execution on EC2 — for example through a supply chain attack where a malicious version of flask or boto3 is published to PyPI and our user data script installs it — they inherit the EC2 instance role. With that role they can scan the entire photoviewer-photos DynamoDB table and read app.py from S3.

The supply chain attack scenario: A malicious actor publishes a backdoored version of boto3 to PyPI. When any server runs pip3 install boto3 it installs the malicious version. The malware runs as root (user data runs as root), inherits the instance role, and exfiltrates data. A real example: in 2018 the npm package event-stream was taken over by a malicious actor who inserted a backdoor targeting cryptocurrency wallets — it was downloaded millions of times before discovery. Package managers are a high-value attack surface.

The fix: We already implemented least privilege — the role only allows dynamodb:Scan on one specific table and s3:GetObject on one specific prefix. The blast radius is contained. Pin package versions in requirements files so pip3 install boto3==1.34.0 rather than whatever is latest. Use CodeArtifact or a private PyPI mirror to control which packages are installable.

No rate limiting on /photos API — Layer 7 DDoS risk
click to expand

The vulnerability: The /photos endpoint has no throttling. A flood of valid HTTP requests — each looking identical to a real user request — would exhaust Flask (single-threaded) and consume DynamoDB read capacity. Static content is naturally protected because CloudFront serves it from cache. Dynamic API responses bypass the cache and hit EC2 every time, making the API the DDoS target.

L3/L4 vs L7 DDoS: Shield Standard on CloudFront protects against volumetric network-layer attacks (SYN floods, UDP amplification) automatically and for free. But application-layer attacks — floods of valid HTTP requests — look identical to real traffic at the network layer. Shield Standard cannot distinguish them. WAF rate limiting is the fix.

The fix: AWS WAF rate-based rule — cap requests per IP per minute on the /photos path. Real users make a handful of requests per session. Attackers make thousands. Rate limiting blocks the attacker while legitimate users are unaffected. We add WAF in Week 9.
