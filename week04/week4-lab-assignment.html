Week 4 Lab — EC2 Backend in VPC

CS 55D: Security in Amazon Web Services  ·  Foothill College  ·  100 pts

Objective: Add a dynamic backend to the Photo Viewer. Replace the static photos.json file with a Flask API server running on EC2 in a private VPC subnet, serving the photo list from DynamoDB. By the end of this lab, photo metadata will be stored in a database and served dynamically through a 3-tier architecture: CloudFront → ALB → EC2 → DynamoDB.

Browser → CloudFront → ALB (public subnet) → EC2 Flask (private subnet) → DynamoDB ↘ S3 (static files via OAC — unchanged from Week 3)
Before you start: Your Week 3 CloudFront distribution must be working. You need your CloudFront domain name and your S3 bucket name ready.
Files for this lab: Get all files from the course GitHub repository → photoviewer/week04

Part 3 — IAM Role

Step 4Create the EC2 IAM role

Go to IAM → Roles → Create role → AWS service → EC2 → Next:

  • Role name: photoviewer-ec2-role → Create role

Add inline policy 1 — DynamoDB read

Click into photoviewer-ec2-role → Add permissions → Create inline policy → JSON tab:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["dynamodb:Scan"],
    "Resource": "arn:aws:dynamodb:us-east-1:[your-account-id]:table/photoviewer-photos"

Policy name: photoviewer-dynamodb-read → Create policy

Part 4 — VPC

Step 5Create VPC with wizard

Go to VPC → Create VPC → select "VPC and more":

  • Name tag: photoviewer
  • IPv4 CIDR: 10.0.0.0/16
  • Number of AZs: 2
  • Number of public subnets: 2
  • Number of private subnets: 2
  • NAT gateways: In 1 AZ
  • VPC endpoints: None

Review the resource map — confirm 2 public subnets, 2 private subnets, 1 IGW, 1 NAT GW → Create VPC. Wait ~2 minutes for all resources to become available.

The wizard creates the VPC, all 4 subnets, the Internet Gateway, route tables, and the NAT Gateway in one shot. The NAT Gateway is temporary — you will delete it in Step 11 after EC2 finishes bootstrapping.
Screenshot: VPC resource map showing subnets, route tables, IGW, and NAT GW all created
Step 6Create security groups

Create 3 security groups in photoviewer-vpc. Create them in this order — SG 2 references SG 1, SG 3 references SG 2.

SG 1 — photoviewer-alb-sg

  • Inbound: HTTP port 80 from 0.0.0.0/0
  • Outbound: delete the default allow-all rule. Add: Custom TCP port 8000, destination: photoviewer-ec2-sg

SG 2 — photoviewer-ec2-sg

  • Inbound: Custom TCP port 8000 from photoviewer-alb-sg
  • Outbound: leave default allow-all
  • No port 22 — SSM Session Manager handles all remote access

SG 3 — photoviewer-endpoint-sg

  • Inbound: HTTPS port 443 from photoviewer-ec2-sg
  • Outbound: leave default allow-all
Screenshot: Each security group showing correct inbound rules
Step 7Create VPC endpoints

Go to VPC → Endpoints → Create endpoint. Create all 5:

Gateway endpoints (free)

  • Name: photoviewer-ep-dynamodb — Service: com.amazonaws.us-east-1.dynamodb — Gateway — associate with private route table
  • Name: photoviewer-ep-s3 — Service: com.amazonaws.us-east-1.s3 — Gateway — associate with private route table

Interface endpoints (SSM — required for Session Manager)

For each: select private subnet in us-east-1a, enable Private DNS, security group: photoviewer-endpoint-sg

  • Name: photoviewer-ep-ssm — Service: com.amazonaws.us-east-1.ssm
  • Name: photoviewer-ep-ssmmessages — Service: com.amazonaws.us-east-1.ssmmessages
  • Name: photoviewer-ep-ec2messages — Service: com.amazonaws.us-east-1.ec2messages

Wait for all 5 endpoints to show status Available.

Screenshot: Endpoints list showing all 5 Available

Part 5 — EC2 instance

Step 8Launch EC2 with user data

Go to EC2 → Launch instance:

  • Name: photoviewer-ec2 — AMI: Amazon Linux 2023 — Instance type: t3.micro
  • Key pair: Proceed without key pair
  • VPC: photoviewer-vpc — Subnet: private subnet in us-east-1a
  • Auto-assign public IP: Disable
  • Security group: photoviewer-ec2-sg
  • IAM instance profile: photoviewer-ec2-role

Under Advanced details → User data, paste the following — replace [your-account-id] with your AWS account ID:

#!/bin/bash
dnf update -y
dnf install -y python3-pip
pip3 install flask boto3
mkdir -p /home/ec2-user/photoviewer
aws s3 cp s3://photoviewer-[your-account-id]/app/app.py /home/ec2-user/photoviewer/app.py
chown -R ec2-user:ec2-user /home/ec2-user/photoviewer
cat > /etc/systemd/system/photoviewer.service << 'SVCEOF'
[Unit]
Description=Photo Viewer App Server
After=network.target
[Service]
User=ec2-user
WorkingDirectory=/home/ec2-user/photoviewer
ExecStart=/usr/bin/python3 app.py
Restart=always
RestartSec=5
[Install]
WantedBy=multi-user.target
SVCEOF
systemctl daemon-reload
systemctl enable photoviewer
systemctl start photoviewer

Click Launch instance. Wait for 2/2 status checks to pass (~3–5 minutes).

Screenshot: EC2 instance running with 2/2 status checks passed
Step 9Verify via SSM Session Manager

Go to EC2 → Instances → select photoviewer-ec2 → Connect → Session Manager tab → Connect.

In the terminal:

systemctl status photoviewer
curl http://localhost:8000/health
curl http://localhost:8000/photos

You should see {"status": "ok"} from health and a JSON array of 5 photos from /photos.

Screenshot: SSM terminal showing systemctl status active and curl /photos returning JSON
Step 10Register EC2 with target group

Go to EC2 → Target Groups → Create target group:

  • Target type: Instances — Name: photoviewer-tg
  • Protocol: HTTP — Port: 8000 — VPC: photoviewer-vpc
  • Health check path: /health → Next
  • Select photoviewer-ec2 → port 8000 → Include as pending → Create target group

Wait for health status to show Healthy (~1 minute).

Screenshot: Target group showing EC2 as Healthy
Step 11Delete NAT Gateway
Do this immediately after Step 10. The NAT Gateway costs ~$32/month. EC2 only needs DynamoDB and SSM at runtime — both go through private endpoints, not the internet.
  1. VPC → NAT Gateways → select the photoviewer NAT gateway → Actions → Delete NAT gateway
  2. VPC → Route tables → select the private route table → Routes → Edit routes → delete the 0.0.0.0/0 → nat-xxxxx entry → Save
  3. Wait ~1 min for NAT GW to show Deleted
  4. EC2 → Elastic IPs → select the unassociated IP → Actions → Release Elastic IP address
Screenshot: Private route table showing no NAT Gateway route

Part 6 — Application Load Balancer

Step 12Create the ALB

Go to EC2 → Load Balancers → Create load balancer → Application Load Balancer:

  • Name: photoviewer-alb — Scheme: Internet-facing
  • VPC: photoviewer-vpc
  • Subnets: select both public subnets (us-east-1a and us-east-1b)
  • Security groups: remove default, add photoviewer-alb-sg
  • Listener: HTTP port 80 → forward to photoviewer-tg
  • Click Create load balancer

Wait for ALB state: Active. Then verify in Target Groups that EC2 still shows Healthy.

Screenshot: ALB showing Active status and DNS name

Part 7 — Wire CloudFront to ALB

Step 13Add ALB as a second CloudFront origin

Go to CloudFront → your distribution → Origins tab → Create origin:

  • Origin domain: paste the ALB DNS name
  • Protocol: HTTP only
  • Name: photoviewer-alb-origin
  • Click Create origin
Step 14Add cache behavior for /photos

Go to Behaviors tab → Create behavior:

  • Path pattern: /photos
  • Origin: photoviewer-alb-origin
  • Viewer protocol policy: Redirect HTTP to HTTPS
  • Cache policy: CachingDisabled
  • Click Create behavior

Go to Invalidations tab → Create invalidation → path: /* → Create invalidation.

Step 15Upload updated app.js

Upload the Week 4 app.js from the GitHub repo to your S3 bucket root, replacing the Week 3 version. Then create another /* CloudFront invalidation.

The only change from Week 3: fetch('/photos.json') becomes fetch('/photos'). The browser now calls the API endpoint on EC2 instead of fetching a static file from S3.

Part 8 — Verify end to end

Step 16Test in browser

Open your CloudFront URL. The Photo Viewer should load photos dynamically from DynamoDB.

Open browser DevTools → Network tab → click the /photos request → Response Headers. Confirm:

  • Server: Werkzeug/x.x.x Python/x.x.x — proves response came from Flask on EC2
  • Via: ... (CloudFront) — proves CloudFront forwarded the request
  • X-Cache: Miss from cloudfront — confirms caching is disabled for API responses
